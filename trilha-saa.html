CAP. 01

IAM e modelo de identidade

Consolidar

A lógica de avaliação de permissões

Toda requisição passa por um "juiz" que avalia todas as policies aplicáveis. A sentença segue esta ordem — sem exceções:

Requisição chega
  → existe Deny explícito?   → SIM → ❌ NEGADO (fim — Deny sempre ganha)
              ↓ não
  → existe Allow explícito?  → SIM → ✅ PERMITIDO
              ↓ não
  → Negação implícita         → ❌ NEGADO (padrão)

Anatomia de uma policy IAM

"Statement": [{
  "Effect":    "Allow",
  "Action":    ["s3:GetObject", "s3:ListBucket"],
  "Resource":  ["arn:aws:s3:::meu-bucket",     ← o BUCKET
               "arn:aws:s3:::meu-bucket/*"],  ← os OBJETOS
  "Condition": { "IpAddress": { "aws:SourceIp": "203.0.113.0/24" }}
}]
⚠
Armadilhaarn:aws:s3:::bucket ≠ arn:aws:s3:::bucket/*. O primeiro é o bucket (ListBucket); o segundo são os objetos (GetObject/PutObject). Esquecer o /* é erro clássico.

Lab 1 — IAM na prática (1h, grátis)
  1. 01

    Crie a role LeitorS3 para EC2 com AmazonS3ReadOnlyAccess, suba uma EC2 com ela e rode aws s3 ls sem credenciais. Depois tente aws s3 cp → observe o AccessDenied.

Checkpoint de revisão

  • Negado. Deny explícito sempre vence qualquer Allow, independente de onde vem.
  • User tem credenciais permanentes e identidade própria. Role é assumida temporariamente via STS — ninguém "é" uma role, apenas a veste por tempo limitado.
  • Evita o "confused deputy problem": garante que apenas o parceiro correto pode assumir a role usando o External ID como segredo compartilhado.
  • No SAA-C03 o desempate costuma ser custo/operação: se o cenário não exige a capacidade extra, a resposta "correta" é a solução mais simples que atende ao requisito — não a mais cara.

Prioridade Alta

Famílias de instância — decore a letra, não o número

O exame raramente pergunta o número exato da instância. Ele descreve um cenário e espera que você reconheça qual família de letra resolve. Memorize o padrão: T/M para uso geral, C para CPU, R/X para memória, I/D para storage local, P/G para GPU.

FamíliaOtimizada paraPalavra-gatilho na questão

CCPU (Compute)"processamento intenso", "batch", "encoding", "HPC"
R / XMemória (RAM)"banco em memória", "SAP", "análise em memória", "muita RAM"
I / DStorage local NVMe"altíssimo IOPS local", "NoSQL auto-gerenciado"

⚠️ Armadilha: instâncias burstable (T) Instâncias da família T acumulam CPUCreditBalance quando ociosas e gastam esses créditos em picos de uso. Se a carga é constante e alta, os créditos se esgotam e a CPU é estrangulada (throttled). Se o enunciado descreve "carga contínua elevada", T é resposta errada — a menos que cite explicitamente o modo unlimited.

🧪 Lab — Compute e custo na prática (1h30, ~US$ 0)
  1. 1
    Suba uma t3.micro e rode stress --cpu 2 por 10 min. Acompanhe no CloudWatch a métrica CPUCreditBalance caindo — você está vendo o conceito burstable ao vivo.
  2. 2
    Crie um volume gp3, anexe, formate e monte. Depois modifique o volume ao vivo (Elastic Volumes) para mais GB e mais IOPS — sem desmontar.
  3. 3
    Tire um snapshot, copie para outra região e restaure lá. Você acaba de executar a operação base de qualquer estratégia de DR.
  4. 4
    Em EC2 → Spot Requests → Pricing History: compare o preço Spot vs On-Demand e internalize o desconto real de 70-90%.

Checkpoint de revisão

  • Spot Instances. Tolerante a interrupção + desconto de até 90%.
  • Dedicated Host — único modelo com visibilidade e controle do hardware físico para licenciamento por socket/core.
  • Somente io1/io2 com Multi-Attach — ou reestruturar para EFS se for armazenamento de arquivo.
  • Os créditos de CPU se esgotaram e a instância está sendo limitada (throttled). Correção: trocar para uma família M/C de uso geral/CPU sem burst, ou ativar o modo "unlimited" da T (cobra excedente por hora).
  • gp3 desacopla IOPS e throughput do tamanho do volume (3.000 IOPS / 125 MB/s de baseline incluídos independente do tamanho) e custa menos por GB que gp2 — sem perda de performance na maioria dos casos.

Storage Gateway — a ponte híbrida

Quando o cenário descreve um datacenter on-premises que precisa continuar acessando dados localmente enquanto eles vivem na nuvem, a resposta é Storage Gateway — não DataSync (que é para migração pontual).

🧪 Lab — Storage completo (2h, centavos)
  1. 1
    Crie um site estático no S3 + lifecycle rule (Standard → IA aos 30d → Glacier aos 90d → expirar aos 365d). Adicione CRR para outra região e veja a replicação.
  2. 2
    Crie um EFS e monte simultaneamente em 2 EC2 em AZs diferentes. Crie um arquivo numa instância e leia na outra.
  3. 3
    Gere uma presigned URL com expiração de 5 min. Teste antes e depois de expirar — confirme o comportamento.
  4. 4
    Configure S3 Event Notification disparando uma Lambda que loga o nome de cada objeto novo no CloudWatch Logs.

Checkpoint de revisão

  • Glacier Deep Archive com lifecycle rule automática. Menor custo de armazenamento do S3.
  • EFS — sistema de arquivos NFS gerenciado, multi-AZ, montado simultaneamente em quantas instâncias precisar.
  • SSE-KMS com Customer Managed Key + CloudTrail. Cada uso da chave gera um evento auditável.
  • Storage Gateway — File Gateway. Mantém o protocolo NFS local com cache, sincronizando os dados de fato com o S3.
  • Quando o enunciado deixa claro que o dado é facilmente recriável (ex.: cópia derivada de outro dado, thumbnail gerado), a resiliência extra de múltiplas AZs é desperdício de custo — One Zone-IA é a opção mais barata que ainda atende ao requisito.
CAP. 04

VPC e Redes

Prioridade Alta

Security Group × NACL — o mantra

Esta é a comparação mais cobrada de redes. Memorize as quatro diferenças abaixo — qualquer questão sobre "bloquear um IP específico" ou "regras de retorno" depende disso.

Security Group

Nível de instância (ENI). Stateful — o tráfego de retorno é permitido automaticamente. Só regras de Allow. Pode referenciar outro SG (ex.: web → app → db).

NACL

Nível de sub-rede. Stateless — é preciso liberar ida e volta explicitamente. Permite Allow e Deny, avaliadas por número de regra. É o único jeito de bloquear um IP específico.

CloudFront × Global Accelerator

CloudFront (CDN)

Faz cache de conteúdo na borda. HTTP/HTTPS. "conteúdo estático global", "vídeo", "reduzir carga na origem". OAC: protege S3 privado servindo apenas via CloudFront.

Global Accelerator

Sem cache — é um atalho de rede pelo backbone AWS. Suporta TCP e UDP. "gaming", "VoIP", "IP estático global", "failover regional em segundos". 2 IPs anycast fixos.

⚠️ Armadilha: NAT Gateway não é Internet Gateway NAT Gateway permite que instâncias em sub-rede privada iniciem conexões para a internet (ex.: baixar updates), mas a internet não pode iniciar conexão com elas. Internet Gateway é o que dá acesso bidirecional às sub-redes públicas. Confundir os dois é erro clássico em questões de arquitetura de VPC.
💡 Sinal prático Se o enunciado menciona tráfego para S3/DynamoDB saindo de sub-rede privada e pede "menor custo", a resposta quase sempre é trocar o caminho via NAT Gateway por um Gateway VPC Endpoint — que é gratuito.
🧪 Lab — Redes (3h, ~US$ 1 — deletar NAT ao final!)
  1. 1
    Recrie a VPC manualmente (2 AZs, sub-redes públicas/privadas, IGW, NAT). Desta vez sem consultar guia — cronometre.
  2. 2
    Adicione um Gateway Endpoint de S3. Remova a rota do NAT e prove que aws s3 ls ainda funciona na EC2 privada — sem internet.
  3. 3
    Crie um CloudFront na frente do bucket com OAC e bloqueie acesso direto. Confirme que só o CloudFront consegue servir os objetos.

Checkpoint de revisão

  • Global Accelerator — TCP/UDP, IPs anycast estáticos, backbone AWS.
  • NACL com regra Deny para o IP. Security Group não tem regra de Deny explícito.
  • Transit Gateway — hub centralizado, evita N*(N-1)/2 peerings.
  • Site-to-Site VPN. Direct Connect oferece um link dedicado e mais estável, mas leva semanas para provisionar — não é a resposta para urgência.
  • Route 53 com routing policy Geolocation. Diferente de Latency, que escolhe pela menor latência e não por requisito legal de país/região.

Aurora — diferenciais que a prova adora

Storage distribuído em 6 cópias por 3 AZs, auto-cura, cresce até 128 TB automaticamente. Até 15 Read Replicas com reader endpoint balanceado. Failover em ~30s. Aurora Global Database: replicação cross-region com lag <1s, promoção em <1 min → RPO ~1s, RTO ~1 min. Resposta padrão para "banco relacional com DR agressivo multi-região".

⚠️ Armadilha: Query vs Scan Query usa a chave de partição (e opcionalmente a de ordenação) para buscar diretamente — rápido e barato. Scan percorre a tabela inteira — lento e caro em tabelas grandes. ElastiCache — Redis × Memcached

Redis

Estruturas ricas (listas, sets, sorted sets). Persistência + replicação + Multi-AZ. "ranking/leaderboard" = sorted sets. Pub/sub, scripts Lua.

Memcached

Cache simples, multi-thread. Sem persistência nem replicação. Dado descartável. Escala horizontal simples.

🧪 Lab — Bancos (2h30, Free Tier)
  1. 1
    Suba RDS MySQL com Multi-AZ. Force um failover (Actions → Reboot with failover) e observe: o endpoint não muda e a conexão volta em ~1 min. Este é o "aha!" do Multi-AZ.
  2. 2
    Crie uma Read Replica, conecte e tente um INSERT → erro de read-only. Promova e veja virar banco independente.
  3. 3
    DynamoDB: tabela pedidos (PK: cliente_id, SK: data). Faça uma Query por cliente e um Scan geral — entenda por que Scan é custoso. Crie um GSI por status.
  4. 4
    Ative TTL num atributo e veja itens expirarem automaticamente.

Checkpoint de revisão

  • Direcionar as queries de relatório para uma Read Replica. O app de escrita continua no primário inalterado.
  • Multi-AZ. Failover automático, mesmo endpoint, sem intervenção.
  • RDS Proxy — faz pool de conexões e as multiplexia, evitando que cada invocação de Lambda abra uma conexão nova.
  • ElastiCache Redis com sorted sets — atualiza e consulta ranking em sub-milissegundo.
  • Provisioned capacity (com auto scaling para folga). On-demand é mais caro por unidade e só se justifica quando a carga é imprevisível.
  • TTL (Time to Live) do DynamoDB — define um atributo de expiração em epoch time e o DynamoDB remove o item automaticamente, sem custo de write/delete.

Desacoplamento — SQS, SNS, EventBridge

Arquiteturas resilientes não fazem componentes chamarem uns aos outros diretamente. Elas inserem uma camada de mensageria que absorve picos e isola falhas. Reconhecer qual peça de mensageria o cenário pede é tão importante quanto saber escalar.

⚠️ Armadilha: SQS — pontos de prova Visibility timeout: a mensagem some da fila enquanto está sendo processada; se não for deletada a tempo, ela reaparece → o consumidor deve ser idempotente. DLQ: após N tentativas com falha, a mensagem vai para a fila de mortos — resposta padrão para "mensagens venenosas". Long polling (20s): reduz custo e chamadas vazias.

  4. 4

Checkpoint de revisão

  • SQS FIFO. Exactly-once, ordem garantida.
  • SNS fan-out para 3 filas SQS, cada uma com seu consumidor (Lambda ou worker). Cada sistema processa de forma independente e assíncrona.
  • Scheduled scaling — escala antes das 9h. Predictive se quiser automatizar a detecção do padrão.
  • HA aceita um pequeno downtime durante o failover (segundos a minutos). FT exige redundância ativa duplicada operando o tempo todo, sem qualquer degradação perceptível — custa significativamente mais e só é exigido quando o enunciado especifica "zero impacto" ou "sem degradação".
  • ALB (Application Load Balancer) — opera na camada 7, suporta WebSocket nativamente e roteamento por path/host-based routing. NLB não entende HTTP.
CAP. 07

Disaster Recovery

Prioridade Alta

RPO e RTO — os dois números que definem tudo

          ←— RPO —→     🔥     ←———— RTO ————→
    último backup    desastre    sistema no ar de novo
    (dados perdidos)            (tempo parado)
💡 Definições RPO (Recovery Point Objective) = quanto de dado posso perder? Olha para o passado — depende de quando foi o último backup/replicação. RTO (Recovery Time Objective) = quanto tempo posso ficar fora do ar? Olha para o futuro — depende de quanto tempo leva para reestabelecer o serviço. Quanto menores os dois, mais cara a estratégia.

As 4 estratégias — da mais barata à mais cara

EstratégiaComo funcionaRPORTOCusto
1. Backup & RestoreSó backups/snapshots copiados p/ outra região. No desastre, reconstrói tudo do zero com CloudFormationHorasHoras-dias💰
2. Pilot LightDados replicando continuamente na região de DR; app desligada mas pronta (AMIs, templates). Ligar os servidores no desastreMinutos1-2 horas💰💰
3. Warm StandbyVersão funcional completa em capacidade mínima rodando na DR. No desastre, redirecionar e escalarSegundos-minMinutos💰💰💰
4. Multi-site Active/ActiveDuas ou mais regiões em produção plena. Desastre = a outra absorve automaticamente~Zero~Zero💰💰💰💰
★ Pilot Light vs Warm Standby — como não confundir Pilot Light: a aplicação NÃO está rodando na DR — só os dados estão vivos. Não atende nenhuma requisição até você "acender". Warm Standby: a aplicação JÁ ESTÁ rodando (mini-versão), mas em capacidade reduzida. Poderia atender uma requisição agora mesmo.

⚠️ Armadilhas de DR Multi-AZ protege contra falha de AZ — não de região inteira. Se "a região ficou indisponível", Multi-AZ não resolve. Snapshots/AMIs precisam ser copiados para outra região — o que ficou na região que falhou não ajuda. "Menor custo com RTO de 24h" → Backup & Restore (não complique desnecessariamente).
🧪 Lab — DR na prática (2h, ~US$ 0,50)

  3. 3

Checkpoint de revisão

  • Pilot Light. Dados replicando continuamente (garante RPO de minutos) + infra desligada (barata). RTO de 1-2h para "acender" a infra é viável.
  • Multi-site Active/Active. Duas ou mais regiões em produção plena, com Route 53 ou Global Accelerator distribuindo o tráfego.
  • Warm Standby. A aplicação já está rodando na DR (distingue de Pilot Light onde ela está desligada).
  • Se toda a região está indisponível, qualquer recurso armazenado nela — incluindo backups e snapshots — também fica inacessível. DR de região exige cópia em outra região (CRR, AWS Backup com cópia cross-region, ou réplicas cross-region).
  • AWS Backup tira snapshots periódicos (RPO em horas). DRS replica continuamente bloco a bloco para a região de DR, permitindo RPO de segundos e RTO de minutos — é o sucessor do CloudEndure para lift-and-shift de servidores inteiros.

  • AWS Batch sobre Fargate — Batch gerencia filas e prioridade, Fargate executa sem gerenciar instâncias. Lambda não serve (limite 15 min).
  • Provisioned Concurrency — mantém instâncias inicializadas e prontas para responder.
  • EKS (Elastic Kubernetes Service). Reaproveita os manifests Kubernetes existentes; ECS exigiria reescrever para task definitions proprietárias.
  • A função deve ser configurada para rodar dentro da VPC (subnets + security groups). Isso permite alcançar o RDS privado, mas remove o acesso padrão à internet pública — se necessário, requer NAT Gateway ou VPC endpoints.
  • Se o requisito é "apenas fazer deploy de um container/código e ter HTTPS, scaling e CI/CD prontos com o mínimo de configuração", App Runner é a resposta. Se há necessidade de controle fino sobre rede, sidecars, ou integração com outros serviços ECS/EKS do ecossistema, use ECS+Fargate.

Prioridade Alta

Chaves simétricas (default) e assimétricas. AWS-managed (grátis, sem controle) vs Customer-managed (CMK) — controle, rotação, policies, auditável via CloudTrail. Chaves são regionais. CloudHSM: hardware dedicado — só quando a questão exige "FIPS 140-2 Level 3" ou "controle exclusivo do hardware".

Rotação automática nativa (RDS!). Pago por segredo/mês. "rotacionar senha do banco automaticamente" = Secrets Manager.

Grátis (standard tier). Configs e segredos com KMS. Sem rotação nativa. "guardar configuração barato" = Parameter Store.

Monitoramento (CloudWatch, CloudTrail, X-Ray)

Consolidar

O trio sagrado — decore o mantra

Três serviços respondem três perguntas diferentes. Quase toda questão de monitoramento se resolve identificando QUAL das três perguntas o enunciado está fazendo.

ServiçoRespondeCasos de uso chave
CloudWatch"O QUE está acontecendo?"Métricas, logs, alarmes, dashboards. Memória da EC2 exige CloudWatch Agent instalado!
CloudTrail"QUEM fez o quê, quando?""Quem deletou o bucket?" Auditoria de API. Trail → S3 para reter além dos 90 dias padrão
Config"COMO está/estava configurado?""Alertar se SG abrir porta 22 ao mundo", histórico de config, remediação automática

X-Ray — rastreando requisições distribuídas

Quando o enunciado descreve uma arquitetura com múltiplos microsserviços (API Gateway → Lambda → DynamoDB → SNS, por exemplo) e pede para identificar onde está o gargalo de latência ou um erro em cadeia, a resposta é AWS X-Ray — ele traça o caminho completo de uma requisição (trace) através dos serviços e mostra um mapa de serviço com tempos de cada salto.

💡 Sinal prático "Identificar qual componente está causando latência numa cadeia de microsserviços" → X-Ray. "O que aconteceu com a CPU/memória/disco?" → CloudWatch. São perguntas diferentes — não confunda observabilidade de uma requisição (X-Ray) com métricas de infraestrutura (CloudWatch).

CloudWatch — componentes essenciais

ComponentePara quê
MetricsSéries temporais numéricas (CPU, requests, latência). Métricas custom via PutMetricData
AlarmsDisparam ação (SNS, Auto Scaling, EC2 actions) quando métrica cruza limiar
Logs / Logs InsightsCentraliza logs de Lambda, ECS, EC2 (via agent). Insights permite consultas tipo SQL nos logs
DashboardsVisualização consolidada de métricas de múltiplos serviços/regiões
Synthetics / Contributor InsightsCanários simulam usuários reais; Contributor Insights identifica top-N contribuintes para um problema
⚠️ Armadilha: CloudWatch não vê dentro da instância por padrão Métricas básicas de EC2 (CPU, rede, status checks) vêm de fora do hypervisor. Métricas de memória RAM e uso de disco exigem o CloudWatch Agent instalado dentro da instância — sem ele, "alertar quando a memória passar de 80%" é impossível.

Analytics (Athena, Glue, Kinesis, Redshift)

Base

O Domínio 3 (alta performance) cobra analytics principalmente como "qual serviço processa/consulta este tipo e volume de dados com menos esforço operacional". A tabela abaixo cobre praticamente todo o vocabulário que aparece nessas questões.

ServiçoUma linha para o SAA
AthenaSQL serverless direto no S3, paga por dado escaneado. Otimize com Parquet + particionamento
GlueETL serverless + Data Catalog (catálogo de metadados usado por Athena/Redshift Spectrum/EMR)
Kinesis Data StreamsIngestão streaming em tempo real, ordenada, com replay. Múltiplos consumidores
Data FirehoseEntrega streaming near-real-time direto p/ S3/Redshift, zero administração
RedshiftData warehouse para BI/OLAP em escala — não para OLTP

Pipeline de streaming — Kinesis Data Streams × Data Firehose

Kinesis Data Streams

Você controla os consumidores (shards, retenção até 365 dias, replay de eventos). Vários consumidores independentes podem ler o mesmo stream. Use quando a aplicação precisa processar os eventos em tempo real com lógica própria.

Data Firehose

Totalmente gerenciado — sem shards para administrar. Entrega automaticamente para S3, Redshift, OpenSearch, com transformação opcional via Lambda. Use quando o destino é só "carregar os dados em algum lugar" sem lógica de consumo customizada.

⚠️ Athena vs Redshift — a distinção favorita "Analisar logs do ALB/VPC Flow que já estão no S3, com SQL, menor custo e esforço" → Athena. Não Redshift, não EMR. Athena é serverless e consulta S3 diretamente sem carregar dado.
💡 Sinal prático "Reduzir o custo de queries no Athena" quase sempre tem a mesma resposta: converter os dados para um formato colunar comprimido (Parquet ou ORC) e particionar por data/categoria — Athena cobra por bytes escaneados, então menos dados lidos = menos custo.

Quando escolher EMR em vez de Glue

Glue é a resposta padrão para ETL serverless simples. EMR entra quando o enunciado menciona frameworks específicos (Spark, Hive, Presto, HBase) com necessidade de configuração detalhada do cluster, jobs de longa duração com estado complexo, ou quando o time já tem expertise em Hadoop/Spark e precisa de controle total sobre o cluster — incluindo o uso de instâncias Spot para os nós de tarefa, reduzindo custo de processamento em até 90%.

  • Data Firehose — totalmente gerenciado, entrega near-real-time para S3/Redshift/OpenSearch, com transformação opcional via Lambda.

  • EMR (Elastic MapReduce) — Hadoop/Spark gerenciado com controle total da configuração do cluster, suportando instâncias Spot nos task nodes para reduzir custo.

OpçãoQuando usar

⚠️ Armadilha: modo de rede awsvpc e limite de ENIs No modo awsvpc (obrigatório no Fargate, recomendado no EC2), cada task recebe sua própria ENI com IP da VPC — isso significa que o número de tasks por instância EC2 fica limitado pelo número de ENIs que o tipo de instância suporta (ENI trunking ajuda, mas é mais um limite a verificar). Em subnets pequenas, muitas tasks Fargate podem esgotar os IPs disponíveis — dimensione a subnet pensando em containers, não só em instâncias EC2.

API Gateway, Step Functions e orquestração serverless

Consolidar

O capítulo 8 cobriu Lambda a fundo, mas tratou API Gateway e orquestração só de passagem. Aqui entramos nos detalhes que separam "funciona no lab" de "passa na prova": tipos de API, controle de tráfego, autenticação, e como coordenar múltiplas Lambdas sem amarrar tudo em código.

REST API vs HTTP API vs WebSocket API

TipoQuando usar
HTTP APIPadrão para APIs REST modernas — até ~70% mais barata e com menor latência que REST API. Suporta JWT authorizers e integração direta com Lambda/HTTP backends
REST APIRecursos completos: API keys, usage plans, transformação de request/response, validação de schema, caching nativo, integração com WAF e endpoints privados via VPC endpoint

Controle de acesso e de tráfego

MecanismoPara quê
Throttling (rate/burst)Limite padrão de conta (token bucket) — protege os backends de picos. Pode ser configurado por método ou por usage plan
Usage Plans + API KeysQuotas e limites por cliente/parceiro — quem está usando o quê, com que limite mensal/diário

IAM AuthorizationChamador assina a requisição com SigV4 — para comunicação entre serviços/contas AWS confiáveis
⚠️ Armadilha: 29 segundos vs 15 minutos O API Gateway tem um timeout fixo de 29 segundos — não configurável — para qualquer integração, mesmo que a Lambda por trás permita até 15 minutos. Se o processamento pode demorar mais que isso, a API Gateway não pode chamar a Lambda de forma síncrona e esperar o resultado. Padrão correto: a API retorna 202 imediatamente (ex.: enfileira em SQS ou inicia uma Step Functions), e o cliente consulta o resultado depois (polling) ou recebe via WebSocket/SNS.

Step Functions: Standard vs Express

Standard Workflows

Duração até 1 ano, execução exactly-once, histórico de execução visual completo, preço por transição de estado. Ideal para workflows de longa duração com aprovações humanas — processos de negócio, orquestração de pipelines.

Express Workflows

Duração até 5 minutos, execução at-least-once, preço por número de execuções + duração/memória (como Lambda). Ideal para alto volume de eventos — processamento de stream, ingestão de dados — com histórico detalhado no CloudWatch Logs.

Agendamento de tarefas

NecessidadeServiço
"Rodar esta Lambda toda noite às 2h"EventBridge Scheduler — expressões cron/rate, fuso horário, alvo flexível (Lambda, Step Functions, SQS, ECS RunTask, APIs de terceiros via conector)
"Reagir quando X acontecer no meu ambiente AWS" (ex.: objeto criado no S3, instância EC2 muda de estado)EventBridge Rules — barramento de eventos com filtro por padrão (event pattern matching)
Milhares de schedules independentes por cliente/tenantEventBridge Scheduler — suporta até milhões de schedules, criados/gerenciados dinamicamente via API

Padrão de orquestração serverless

// Cliente chama API Gateway → inicia Step Functions (assíncrono, evita o timeout de 29s)
Cliente → API Gateway → StartExecution("PipelineProcessamento")
                        ↓
              Step Functions (Standard)
                ├─ Lambda: valida entrada
                ├─ Lambda: chama serviço de IA (cap. 18)
                ├─ Choice: sucesso? → próximo : Catch → notifica SNS
                └─ Lambda: grava resultado no DynamoDB
★ Quando Step Functions substitui "Lambda chamando Lambda" Se você está tentado a fazer uma Lambda invocar outra Lambda de forma síncrona (e mais outra, e mais outra), pare: isso cria acoplamento forte, dificulta debug e pode gerar custo duplicado de espera. Step Functions move a lógica de "o que vem depois" para fora do código — cada Lambda fica pequena e independente, com retries, timeouts e tratamento de erro configurados visualmente no workflow.

  2. 2

Checkpoint de revisão

  • HTTP API — mais barata e com menor latência que REST API, suficiente quando não há necessidade dos recursos avançados (usage plans, caching, transformação de payload).
  • Standard Workflow — suporta execuções de até 1 ano, com estados de espera (ex.: "Wait for Task Token") até alguém aprovar manualmente.
  • Express Workflow — preço por execução/duração compensa em alto volume, e a semântica at-least-once é aceitável para esse tipo de processamento.
  • O API Gateway tem timeout fixo de 29s, independente do timeout da Lambda (até 15 min). Corrigir tornando a chamada assíncrona: a API retorna 202 + um id de job (ex.: inicia Step Functions ou enfileira em SQS), e o cliente consulta o status depois.
  • REST API com endpoint do tipo Private, associado a um VPC endpoint (Interface Endpoint) — o tráfego nunca sai para a internet pública, e uma resource policy controla quais VPCs/contas podem chamá-la.

O capítulo 4 já cobriu as políticas de roteamento do Route 53 e a comparação CloudFront vs Global Accelerator. Aqui vamos além: tipos de registro, health checks compostos, e os detalhes de cache/segurança do CloudFront que aparecem em questões sobre distribuição de conteúdo privado e performance global.

Tipos de registro: Alias vs CNAME

RegistroCaracterística
Alias (A/AAAA)Extensão da AWS — aponta para recursos AWS (ALB, CloudFront, S3 website, outro registro Route 53) de graça (sem cobrança de consulta) e funciona no apex da zona (ex.: exemplo.com, sem subdomínio)

Health checks do Route 53

TipoComo funciona
Basic (endpoint)Verifica um IP/domínio diretamente — HTTP/HTTPS/TCP — com checadores globais distribuídos avaliando a % de respostas saudáveis
CalculatedCombina o status de até 256 outros health checks com lógica AND/OR/NOT — ex.: "saudável se pelo menos 2 de 3 regiões estiverem de pé"
CloudWatch AlarmReflete o estado de um alarme do CloudWatch — útil para monitorar recursos que o Route 53 não alcança diretamente (ex.: RDS, métricas de aplicação)
★ Failover automático = Health Check + Routing Policy Failover Health checks por si só não redirecionam tráfego — eles apenas marcam um registro como saudável/não saudável. É a política de roteamento Failover (cap. 4) que usa esse status para decidir entre o registro primário e o secundário. Sempre que o enunciado pedir "DNS muda automaticamente se a região cair", são as duas peças juntas.

CloudFront: cache e invalidação

ConceitoO que controla
Cache behaviorRegras por path pattern (/imagens/*, /api/*) — origem, TTL, política de cache e de headers/cookies/query strings encaminhados
TTL (min/default/max)Quanto tempo o objeto fica no edge antes de revalidar com a origem — pode ser sobrescrito pelos headers Cache-Control/Expires da origem
InvalidaçãoForça a remoção de objetos do cache antes do TTL expirar — primeiros 1.000 paths/mês grátis, depois cobrado por path
Versionamento de objetoAlternativa à invalidação: mudar o nome do arquivo (app.v2.js) — evita custo de invalidação e propaga instantaneamente

Acesso privado ao S3: OAC vs OAI

OAC (Origin Access Control) — atual

Suporta SSE-KMS no bucket, todos os métodos HTTP (incluindo POST/PUT para uploads via CloudFront) e assinatura SigV4 completa. É a opção recomendada pela AWS para qualquer distribuição nova.

OAI (Origin Access Identity) — legado

Ainda funciona e aparece em ambientes antigos/questões de prova, mas não suporta SSE-KMS nem todos os métodos. Se a questão menciona bucket criptografado com KMS, OAI não atende — precisa de OAC.

Conteúdo restrito: Signed URLs vs Signed Cookies

CenárioEscolha
Acesso a um único arquivo (ex.: link de download de um PDF)Signed URL — query string com assinatura e expiração, válida para aquele objeto/path
Acesso a múltiplos arquivos (ex.: todos os vídeos de um curso após login)Signed Cookies — o navegador envia o cookie em toda requisição subsequente, sem precisar assinar cada URL
Restringir por geografia/paísGeo Restriction da distribuição (allowlist/denylist por país) — não precisa de signed URL/cookie

Personalização no edge: Lambda@Edge vs CloudFront Functions

CloudFront Functions

JavaScript leve, execução em microssegundos, milhões de requisições/segundo, roda apenas em viewer request/response. Ideal para manipulações simples: reescrever URLs, normalizar headers, checagens de autorização leves.

Lambda@Edge

Funções Lambda (Node.js/Python) com mais recursos e tempo de execução, podem rodar nas 4 fases (viewer/origin request/response) e acessar rede externa. Use quando precisar de lógica mais pesada — ex.: chamar uma API externa antes de servir o conteúdo.

  • Signed Cookies — um conjunto de cookies assinados é emitido após o login e enviado automaticamente pelo navegador em todas as requisições subsequentes para qualquer objeto autorizado, sem precisar gerar uma URL assinada por arquivo.

Redes avançadas e conectividade híbrida

Base

O capítulo 4 introduziu Transit Gateway, VPN e Direct Connect na tabela de "conectividade". Aqui aprofundamos cada um — especialmente os cenários de múltiplas VPCs/contas/regiões e os detalhes de redundância que o exame cobra em questões de "arquitetura corporativa".

Transit Gateway em profundidade

O TGW é um "roteador regional" central: cada VPC/VPN/Direct Connect se conecta a ele via um attachment, e route tables do TGW (não as route tables da VPC) decidem quem enxerga quem.

RecursoDetalhe
TGW Route TablesCada attachment é associado a uma route table do TGW — permite segmentar (ex.: VPCs de produção não veem VPCs de dev, mesmo conectadas ao mesmo TGW)
Peering entre TGWs (inter-region)Conecta TGWs de regiões diferentes — tráfego permanece na rede backbone da AWS, criptografado
MulticastTGW suporta grupos multicast entre subnets de VPCs anexadas — caso de uso de nicho, mas que aparece como "diferencial" do TGW
Compartilhamento via RAMUm TGW criado na conta de rede central pode ser compartilhado com outras contas da Organization
⚠️ Armadilha: VPC Peering não é transitivo, Transit Gateway é Se a VPC A está peered com B, e B está peered com C, A não consegue alcançar C através de B — cada peering é um túnel ponto a ponto isolado. Para "dezenas de VPCs precisam se comunicar entre si" sem criar uma malha de N×(N-1)/2 peerings, a resposta é Transit Gateway (hub-and-spoke), que roteia entre todos os attachments conforme as route tables do TGW permitirem.

Direct Connect: tipos de interface virtual (VIF)

TipoPara quê
Private VIFAcessa recursos dentro de uma VPC (via Virtual Private Gateway ou Direct Connect Gateway) — IPs privados, como se estivesse na mesma rede
Public VIFAcessa serviços públicos AWS (S3, DynamoDB, etc.) usando seus endpoints públicos, mas pela conexão dedicada — sem passar pela internet
Transit VIFConecta a um Direct Connect Gateway associado a um ou mais Transit Gateways — para alcançar múltiplas VPCs/regiões por uma única conexão física
LAG (Link Aggregation Group)Agrupa múltiplas conexões DX físicas em uma só conexão lógica — mais throughput e redundância física
Direct Connect GatewayPermite que uma conexão DX alcance VPCs em múltiplas regiões (exceto China) através de Private/Transit VIFs

VPN: redundância e performance

ConceitoDetalhe
2 túneis por Site-to-Site VPNCada conexão VPN vem com 2 túneis IPsec para redundância — o ideal é configurar o roteador on-premises para usar ambos (ECMP ou BGP com prioridades)
BGP (dinâmico) vs estáticoBGP permite failover automático de rotas entre os túneis e detecta o caminho mais curto. Roteamento estático exige atualização manual das rotas
VPN acelerada (Global Accelerator)Usa a rede global da AWS para reduzir latência até o endpoint VPN — útil quando o caminho de internet até a região é ruim
VPN como backup do Direct ConnectPadrão comum: DX como link primário (baixa latência, alta largura de banda) + VPN Site-to-Site como failover automático via BGP

   ↓ aprovação manual ou automática de cada subscrição
// VPC do cliente (consumidor)
Interface Endpoint → ENI privada na subnet do cliente → acessa seu serviço por IP privado
💡 Quando usar PrivateLink em vez de Peering Peering expõe toda a CIDR da VPC ao par. PrivateLink expõe apenas o serviço específico atrás do NLB — sem rotas de VPC, sem risco de sobreposição de CIDR entre as VPCs, e com controle granular de quem pode se conectar. Para "SaaS multi-tenant" ou "compartilhar um serviço com parceiros sem expor a rede inteira", a resposta é PrivateLink.

VPC Sharing (AWS RAM)

O AWS Resource Access Manager permite que uma conta "proprietária" compartilhe subnets de sua VPC com outras contas da Organization. As contas participantes podem lançar recursos (EC2, RDS, Lambda em VPC) nessas subnets compartilhadas, mas não podem gerenciar a VPC em si — útil para times de aplicação que não devem administrar a rede, mantendo um único bloco CIDR/Transit Gateway centralizado.

🧪 Lab — Transit Gateway entre duas VPCs (1h, ~US$ 0,10)
  1. 1
    Crie duas VPCs em CIDRs diferentes (ex.: 10.0.0.0/16 e 10.1.0.0/16), cada uma com uma subnet e uma instância EC2 (sem IP público).
  2. 2
    Crie um Transit Gateway e anexe as duas VPCs (um attachment por VPC).
  3. 3
    Atualize as route tables de cada VPC para rotear o CIDR da outra VPC através do TGW, e confirme que a route table do TGW propaga ambos os attachments.

Checkpoint de revisão

  • Transit Gateway com múltiplas route tables — cada departamento associado a uma route table que só propaga as rotas permitidas, em vez de uma malha de VPC peerings (que seria não-transitiva e inviável em escala).
  • Direct Connect Gateway associado a Transit VIFs (conectados a Transit Gateways em cada região) — o DX Gateway estende o alcance da conexão física para múltiplas regiões.
  • Site-to-Site VPN como backup, com BGP configurado para assumir automaticamente as rotas quando o DX cair — ambos anunciam as mesmas rotas, e o roteador escolhe o caminho com menor custo BGP (DX) enquanto disponível.
  • AWS PrivateLink (VPC Endpoint Service na frente de um NLB) — expõe apenas o serviço específico via ENI privada na VPC do cliente, sem peering e sem expor a rede inteira.
  • AWS Network Firewall com regras de filtragem por domínio (stateful), posicionado na rota de saída da VPC — SG/NACL não filtram por FQDN, e WAF só atua em tráfego HTTP de entrada.

Machine Learning e IA na AWS

Base

O SAA-C03 não pede para você treinar modelos — pede para você escolher o serviço certo entre três níveis de abstração: SageMaker (construir do zero), Bedrock (usar modelos de fundação prontos) e serviços de IA pré-treinados (uma API para uma tarefa específica). Errar o nível é o erro mais comum nessas questões.

Amazon Bedrock e o padrão RAG

O Bedrock dá acesso via API a modelos de fundação (foundation models) de múltiplos provedores (Anthropic, Meta, Amazon Titan, etc.) sem provisionar GPU. O padrão mais cobrado é RAG (Retrieval-Augmented Generation):

// Pergunta do usuário não vai direto pro modelo — primeiro busca contexto relevante
Usuário → pergunta
   ↓
Knowledge Base (Bedrock) → busca vetorial nos seus documentos (S3 + OpenSearch/Aurora pgvector)
   ↓
Foundation Model ← pergunta + trechos relevantes recuperados
   ↓
resposta fundamentada nos SEUS dados, sem precisar re-treinar o modelo
★ RAG vs Fine-tuning Se o enunciado diz "as respostas devem refletir documentos internos atualizados frequentemente" — RAG (Knowledge Bases for Bedrock), porque os dados ficam fora do modelo e são buscados em tempo real. Fine-tuning (treinar o modelo com dados próprios) é mais caro, mais lento de atualizar, e raramente é "a resposta certa" quando a palavra-chave é "atualizado"/"dinâmico".

  • Amazon SageMaker — controle total do ciclo de vida (preparação de dados, treino, tuning, deploy) para um modelo customizado treinado com dados próprios.
  • Serverless Inference — escala a zero quando não há tráfego, cobrando apenas pelo tempo de computação usado, aceitando uma cold start adicional na primeira chamada após período ocioso.
  • Amazon Transcribe primeiro (áudio → texto), depois Amazon Comprehend sobre o texto resultante (análise de sentimento) — Comprehend não processa áudio diretamente.
CAP. 19

AWS Organizations, Control Tower e governança multi-conta

Consolidar

Empresas reais raramente vivem em uma única conta AWS — usam múltiplas contas para isolar produção de desenvolvimento, separar times e limitar o "blast radius" de erros e incidentes de segurança. Este capítulo cobre como a AWS organiza, governa e fatura essas contas como um todo.

Estrutura do AWS Organizations

ConceitoO que é
Management accountA conta "raiz" da Organization — paga a fatura consolidada e cria/gerencia as demais contas. Não deve hospedar workloads (best practice)
Member accountsContas individuais (produção, dev, segurança, logging...) — isoladas entre si por padrão, cada uma com seu próprio espaço de recursos e IAM
Organizational Units (OUs)"Pastas" que agrupam contas com necessidades de política semelhantes (ex.: OU "Produção", OU "Sandbox") — SCPs aplicadas em uma OU são herdadas por todas as contas dentro dela
RootO nível mais alto da árvore, contém todas as OUs/contas — SCPs aplicadas aqui afetam toda a Organization

Service Control Policies (SCPs)

SCPs definem o teto máximo de permissões para todas as identidades (usuários e roles, incluindo administradores) dentro das contas afetadas — mas, diferente de uma política IAM, uma SCP nunca concede nada por si só.

// SCP "deny-list": bloqueia uma ação específica e herda o resto normalmente
{
  "Effect": "Deny",
  "Action": "ec2:*",
  "Resource": "*",
  "Condition": { "StringNotEquals": { "aws:RequestedRegion": ["sa-east-1"] } }
}
// resultado: ninguém na conta/OU usa EC2 fora de São Paulo — nem mesmo um admin com AdministratorAccess
⚠️ Armadilha: SCP define o teto, não concede acesso Uma SCP do tipo allow-list aplicada no Root (ex.: "Effect": "Allow", "Action": ["s3:*", "ec2:*"]) substitui a política padrão "FullAWSAccess" — a partir daí, nenhuma ação fora dessa lista funciona em conta alguma, mesmo que a política IAM do usuário permita. SCP e política IAM/permission boundary (cap. 1) precisam ambas permitir para a ação ser executada: é uma intersecção de permissões, não uma soma.

AWS RAM e estratégia de tags

AWS Resource Access Manager (RAM) compartilha recursos específicos — subnets de VPC, Transit Gateways, licenças, regras do Route 53 Resolver — entre contas da Organization, sem duplicar o recurso nem montar peering/cross-account IAM para cada caso. Combinado com uma estratégia de tags consistente (ex.: cost-center, environment, owner aplicadas via Tag Policies do Organizations) e Cost Allocation Tags ativadas no Billing, a empresa consegue fatiar o Cost & Usage Report (cap. 13) por equipe ou projeto mesmo dentro de uma fatura consolidada única.

  • Sim, por padrão. Savings Plans (assim como RIs) compartilham o desconto com todas as contas da Organization que tiverem consolidated billing habilitado e uso elegível, a menos que esse compartilhamento seja explicitamente desligado.
  • AWS Resource Access Manager (RAM) — compartilha subnets específicas da VPC com as contas da Organization; as contas participantes lançam recursos nessas subnets, mas a VPC continua gerenciada centralmente.
CAP. 20

AWS Well-Architected Framework

Base

Este é o capítulo "guarda-chuva" da prova: o Well-Architected Framework não introduz serviços novos, mas dá uma lente de leitura para qualquer questão de cenário. Quando o enunciado descreve uma arquitetura e pergunta "qual a MELHOR solução", ele está, na prática, pedindo para identificar qual pilar está sendo testado — e a resposta certa é quase sempre a que resolve esse pilar sem sacrificar demais os outros.

Os seis pilares

PilarFocoPergunta-chave
Operational ExcellenceOperar e monitorar sistemas, melhorando processos continuamente"Como saberemos que algo quebrou, e como automatizamos a resposta?"
SecurityProteger dados, sistemas e ativos via gestão de risco"Quem pode acessar o quê, e os dados estão protegidos em repouso/trânsito?"
ReliabilityRecuperação de falhas, escalonamento dinâmico de demanda"O sistema se recupera sozinho de uma falha de AZ ou instância?"
Performance EfficiencyUso eficiente de recursos computacionais conforme a demanda evolui"Estamos usando o tipo certo de recurso para essa carga de trabalho?"
Cost OptimizationEvitar gastos desnecessários e entender para onde o dinheiro vai"Estamos pagando por capacidade ociosa quando algo serverless/Spot resolveria?"
SustainabilityMinimizar os impactos ambientais das workloads em nuvem"Podemos reduzir consumo de energia escolhendo regiões/instâncias mais eficientes?"
🧠 Mantra: a palavra-chave do enunciado aponta o pilar "Mínimo esforço operacional" → Operational Excellence (pense gerenciado/serverless). "Criptografado", "least privilege", "auditoria" → Security. "Sobrevive à perda de uma AZ", "RPO/RTO" → Reliability. "Mais rápido", "menor latência" → Performance Efficiency. "Mais barato", "reduzir custo" → Cost Optimization. Quando duas opções parecem tecnicamente corretas, o pilar destacado no enunciado decide o desempate.

  • Operational Excellence. "Mínimo esforço operacional" é a palavra-chave clássica para soluções serverless/gerenciadas — Lambda + API Gateway tende a ser a resposta, pois elimina patching, capacity planning e gestão de servidores.
  • Cost Optimization. Para workloads tolerantes a interrupção, Spot Instances vencem por custo significativamente menor — RIs fazem sentido para cargas previsíveis e contínuas, não para batch jobs flexíveis.
  • AWS Well-Architected Tool — questionário guiado por workload, sem custo adicional, disponível em qualquer plano de suporte.
  • Não. No plano Basic/Developer, o Trusted Advisor expõe apenas um subconjunto de checks de Security e Service Limits. Cobertura completa das 5 categorias exige plano Business ou Enterprise.
  • Reliability. RPO/RTO numéricos sinalizam que a resposta precisa de uma estratégia de DR concreta (cap. 7) — Multi-AZ ou Multi-Region com backups/replicação que atendam a essas metas, não apenas "fazer backup".
  • Sustainability. Ações típicas: escolher regiões com matriz energética mais limpa, usar instâncias mais eficientes (ex.: Graviton), desligar recursos ociosos e adotar arquiteturas serverless que escalam a zero.
