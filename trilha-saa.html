CAP. 01

IAM — Identidade, Acesso e Federação

Consolidar

O IAM (Identity and Access Management) é o sistema nervoso de segurança de toda conta AWS. Cada chamada de API que você faz passa pelo IAM antes de chegar ao serviço destino. Entender IAM profundamente é pré-requisito para todos os outros capítulos — especialmente os de segurança, rede e arquitetura multi-conta.

A lógica de avaliação de permissões

Toda requisição passa por um "juiz" que avalia todas as policies aplicáveis. A ordem de avaliação é determinística e não tem exceções:

Requisição chega
  → 1. existe Deny explícito em QUALQUER policy?  → SIM → ❌ NEGADO (fim)
              ↓ não
  → 2. existe Allow explícito em alguma policy?   → SIM → ✅ PERMITIDO
              ↓ não
  → 3. Negação implícita                          → ❌ NEGADO (padrão)
⚠
Acesso cross-account é diferente — na mesma conta, um Allow em qualquer policy é suficiente. Em acesso cross-account, AMBAS as contas precisam permitir: a identity policy da conta de origem E a resource-based policy (ou role trust policy) da conta de destino. Um único Allow não basta.

Anatomia de uma policy IAM

Uma policy é um documento JSON com um ou mais Statements. Cada statement tem cinco campos principais:

"Statement": [{
  "Sid":       "PermitirLeituraS3",        ← identificador opcional (bom para documentação)
  "Effect":    "Allow",                    ← Allow ou Deny
  "Principal": "*",                       ← quem? Só em resource-based policies
  "Action":    ["s3:GetObject", "s3:ListBucket"],
  "Resource":  ["arn:aws:s3:::meu-bucket",     ← o BUCKET (para ListBucket)
               "arn:aws:s3:::meu-bucket/*"],  ← os OBJETOS (para GetObject)
  "Condition": {
    "IpAddress":       { "aws:SourceIp": "203.0.113.0/24" },
    "StringEquals":    { "aws:RequestedRegion": "us-east-1" },
    "Bool":            { "aws:MultiFactorAuthPresent": "true" }
  }
}]
⚠
Armadilha clássica de ARN: arn:aws:s3:::bucket identifica o bucket em si (necessário para ListBucket). arn:aws:s3:::bucket/* identifica os objetos dentro do bucket (necessário para GetObject/PutObject). Esquecer qualquer um dos dois gera AccessDenied — e essa distinção cai muito no exame.

STS — o motor das credenciais temporárias

O AWS Security Token Service (STS) é o serviço que emite credenciais temporárias (Access Key + Secret Key + Session Token + Expiration). Toda vez que algo "assume" uma role, está chamando o STS. As credenciais expiram entre 15 minutos e 12 horas, dependendo da configuração.

Cross-account access — como funciona na prática

Cenário: Conta A (onde está o desenvolvedor) quer acessar recursos na Conta B (onde está o bucket S3).

1. Na Conta B: crie uma Role com trust policy confiando na Conta A
{
  "Principal": { "AWS": "arn:aws:iam::111111111111:root" },  ← Conta A
  "Action": "sts:AssumeRole",
  "Condition": { "StringEquals": { "sts:ExternalId": "segredo-compartilhado" }}
}

2. Na Conta A: permita que o usuário/role assuma a role da Conta B
{ "Action": "sts:AssumeRole", "Resource": "arn:aws:iam::222222222222:role/RoleNaContaB" }

3. O usuário na Conta A chama:
aws sts assume-role --role-arn arn:aws:iam::222222222222:role/RoleNaContaB \
                   --external-id segredo-compartilhado \
                   --role-session-name minha-sessao
💡
External ID evita o "confused deputy problem" — se uma ferramenta de terceiro (ex.: empresa de auditoria) usa a sua role, um atacante que comprometesse outra conta cliente dessa ferramenta poderia fingir ser a sua empresa. Com External ID (segredo exclusivo do par fornecedor+cliente), isso não é possível.

Permission Boundary — entendendo a interseção

Um Permission Boundary define o teto máximo do que uma identity pode fazer, independente de qualquer policy concedida. O acesso efetivo é a interseção entre a identity policy e o boundary — não a soma.

Identity policy permite: s3:*, ec2:*, iam:*
Permission boundary permite: s3:*, ec2:*
Resultado: s3:*, ec2:* — iam:* é cortado pelo boundary

Uso principal: permitir que desenvolvedores criem roles para suas aplicações, mas impedir que criem roles com mais permissão do que eles mesmos têm. O dev tem um boundary que inclui apenas os serviços que usa — qualquer role que ele criar herda esse teto.

ABAC — controle baseado em atributos (tags)

ABAC (Attribute-Based Access Control) usa tags como condições nas policies, eliminando a necessidade de criar uma policy por recurso. Em vez de "developer-joao pode acessar o bucket-joao", você escreve: "qualquer desenvolvedor pode acessar qualquer recurso com a mesma tag de equipe".

Policy ABAC: "Acesse recursos com a mesma tag de equipe que a sua identidade"
{
  "Condition": {
    "StringEquals": {
      "aws:ResourceTag/Equipe": "${aws:PrincipalTag/Equipe}"
    }
  }
}

Vantagem: escala automaticamente — ao adicionar um novo membro à equipe, basta dar a tag correta. Sem ABAC, seria necessário atualizar a policy.

IAM Access Analyzer — descubra o que está exposto

O IAM Access Analyzer analisa políticas e identifica recursos que estão acessíveis a entidades externas à sua zona de confiança (conta ou organização). Emite findings para:

  • Buckets S3 com acesso público ou cross-account
  • Roles com trust policies que permitem acesso de contas externas
  • KMS keys, Lambda functions, SQS queues, Secrets Manager secrets
  • Policies excessivamente permissivas (geração de policies com princípio de menor privilégio)

Políticas de senha e MFA

O IAM permite configurar uma Account Password Policy que se aplica a todos os usuários IAM da conta: comprimento mínimo, complexidade, histórico de senhas, expiração. Para o root e usuários privilegiados, MFA é obrigatório como prática de segurança — e o exame frequentemente exige que você saiba que o root não pode ter policies IAM aplicadas a ele (ele tem acesso total por definição).

💡
Sinal prático para o exame: Quando o enunciado fala em "milhões de usuários" ou "app mobile" → Cognito. "Funcionários da empresa com AD" → Identity Center ou SAML. "Serviço AWS acessando outro" → Role. "Chave de acesso no código" → nunca — sempre role ou Secrets Manager.

Checkpoint de revisão

  • Negado. Deny explícito sempre vence qualquer Allow, independente de onde vem — seja em identity policy, resource policy, SCP, ou qualquer outro tipo.
  • User tem credenciais permanentes (access key / senha) e identidade própria. Role é assumida temporariamente via STS — ninguém "é" uma role, apenas a veste por tempo limitado. Credenciais da role expiram (15 min a 12h). Roles não têm senhas nem access keys estáticas.
  • Evita o "confused deputy problem": quando um fornecedor gerencia múltiplos clientes, sem External ID uma conta cliente comprometida poderia usar o fornecedor para acessar a sua conta. Com External ID (segredo exclusivo do par fornecedor+você), o fornecedor só assume a sua role se apresentar o ID correto.
  • User Pool autentica usuários e emite JWTs (quem você é). Identity Pool troca um token de autenticação (JWT do User Pool, Google, Facebook, SAML) por credenciais AWS temporárias via STS (o que você pode fazer na AWS). User Pool = porteiro. Identity Pool = crachá temporário de acesso à AWS.
  • Apenas s3:GetObject. Permission boundary define o teto — o acesso efetivo é a INTERSEÇÃO entre a identity policy e o boundary. Mesmo com AdministratorAccess na policy, o boundary limita a apenas o que ambos permitem simultaneamente.
  • Instance Profile (IAM Role na EC2) para autenticação + VPC Endpoint para que o tráfego não saia para a internet. A EC2 obtém credenciais temporárias via IMDS na rede local, e o VPC Endpoint roteia chamadas ao serviço AWS (ex.: S3, STS) pela rede privada da AWS.
CAP. 02

Computação — EC2, Auto Scaling e ELB

Prioridade Alta

EC2 é o serviço de computação base da AWS e aparece em quase toda arquitetura. O SAA-C03 vai muito além de "suba uma instância" — ele testa sua capacidade de escolher o modelo de compra certo, dimensionar corretamente com Auto Scaling, e distribuir carga com o tipo de Load Balancer adequado para cada protocolo.

Famílias de instância — decore a letra, não o número

O exame raramente pergunta o tipo exato da instância. Ele descreve um cenário e espera que você reconheça qual família resolve. Memorize o padrão abaixo:

FamíliaOtimizada paraPalavra-gatilho no enunciado
T (burstable)Uso geral com créditos de CPU"desenvolvimento", "carga variável", "baixo custo com picos ocasionais"
M (geral)Uso geral balanceado"web server de produção", "microserviço", sem requisito extremo
C (compute)CPU intenso"processamento intenso", "encoding de vídeo", "HPC", "análise científica"
R / XMemória (RAM)"banco em memória", "SAP HANA", "Redis", "enorme conjunto de dados em RAM"
I / DStorage local NVMe"altíssimo IOPS local", "NoSQL auto-gerenciado", "Cassandra sem EBS"
P / G / Trn / InfGPU / Aceleradores"treinamento de ML", "inferência", "renderização", "deep learning"
⚠
Armadilha: instâncias burstable (T) — A família T acumula CPUCreditBalance quando ociosa e gasta créditos em picos. Se a carga é constante e alta, os créditos se esgotam e a CPU fica throttled. Se o enunciado diz "carga contínua elevada" e você ver T3/T3a como opção, é distrator. A correção é migrar para M (balanceado) ou C (CPU), ou ativar o modo unlimited da família T (que cobra o excedente por hora).

User Data e Instance Metadata Service (IMDS)

O User Data é um script executado uma única vez durante o primeiro boot da instância (como root). É o mecanismo padrão para instalar pacotes, puxar código do S3, configurar serviços, etc. Para depurar, veja os logs em /var/log/cloud-init-output.log.

O IMDS (Instance Metadata Service) é acessível de dentro da instância pelo endpoint 169.254.169.254. Ele fornece metadados da instância (ID, tipo, AZ, IP), e mais importante: as credenciais temporárias da Instance Profile (Role). O SDK da AWS busca credenciais aí automaticamente — por isso a EC2 não precisa de access keys no código.

💡
IMDSv2 vs IMDSv1 — A versão v2 do IMDS exige um token temporário antes de aceitar requisições, protegendo contra ataques SSRF que tentam roubar credenciais via vulnerabilidades na aplicação. No exame, se a questão falar em "proteção contra SSRF para roubo de credenciais EC2", a resposta é habilitar IMDSv2 obrigatório.

EBS — volumes de armazenamento em bloco

O EBS (Elastic Block Store) é o disco padrão da EC2. Persiste independente da instância (ao contrário do Instance Store que some quando a EC2 para). Um volume EBS só pode ser anexado a uma instância por vez — exceto io1/io2 com Multi-Attach. O EBS fica na mesma AZ que a instância.

Snapshots EBS são backups incrementais armazenados em S3 (mas você não os vê no console do S3). Podem ser copiados entre regiões — operação básica em qualquer estratégia de DR. O Data Lifecycle Manager automatiza criação, retenção e exclusão de snapshots.

Instance Store — armazenamento efêmero de alta performance

O Instance Store é armazenamento físico diretamente acoplado ao servidor (NVMe). Desaparece se a instância parar, falhar ou for terminada. Não pode ser detachado nem movido. Fornece o maior IOPS possível na AWS (milhões de IOPS em instâncias i4i/i3en).

⚠
Não persiste dados — Use Instance Store apenas para dados temporários que podem ser recriados: buffers, caches, scratch space, dados que replicam do cluster (ex.: Cassandra). Nunca para dados únicos ou que precisam sobreviver a uma falha de instância.

Health Checks do ASG

Por padrão, o ASG usa apenas o status do EC2 (EC2 health check). Para substituir instâncias que estão de pé mas servindo erros HTTP, configure o ASG para usar o ELB health check — o load balancer determina se a instância está saudável. Se não estiver, o ASG termina a instância e sobe uma nova.

Elastic Load Balancer (ELB) — tipos e quando usar cada um

O ELB distribui tráfego entre múltiplas instâncias, containers ou IPs. Existem 4 tipos — a escolha certa no exame depende do protocolo e dos requisitos do cenário:

NLB (Network)L4TCP, UDP, TLSUltra-baixa latência, IP estático, tráfego não-HTTP (jogos, IoT), necessita IP fixo por AZ

ALB — roteamento avançado e recursos de L7

O ALB é o mais cobrado no exame. Recursos que aparecem frequentemente:

  • Roteamento baseado em path: /api/* → Target Group A (API), /static/* → Target Group B (servidores de conteúdo).
  • Roteamento baseado em host: api.exemplo.com → TG diferente de www.exemplo.com.
  • Roteamento baseado em cabeçalhos/query strings: direcionar usuários beta para versão canary.
  • Autenticação integrada: suporta Cognito User Pool e provedores OIDC — elimina código de auth na aplicação.
  • Target Groups heterogêneos: pode incluir EC2, ECS tasks, Lambda, IPs privados, e até outro ALB.
  • Sticky Sessions: cookies de afinidade para manter usuário na mesma instância — útil para sessões com estado local (mas evite em favor de sessões centralizadas em Redis/DynamoDB).

NLB — quando IP estático e latência mínima importam

O NLB opera na camada 4 — ele não lê HTTP, apenas encaminha pacotes TCP/UDP. Como consequência:

  • Latência muito menor (microssegundos vs milissegundos do ALB).
  • IP estático por AZ — essencial quando clientes precisam colocar IPs em whitelist.
  • Preserva o IP de origem do cliente (o ALB substitui pelo IP do load balancer; para recuperar o IP original no ALB, use o header X-Forwarded-For).
  • Suporta integração com AWS PrivateLink para expor serviços internos a outras VPCs de forma privada.

Cross-Zone Load Balancing

Por padrão, um ELB distribui tráfego igualmente entre os nós do load balancer em cada AZ — não entre as instâncias. Se você tem 1 instância na AZ-A e 9 na AZ-B, a AZ-A pode receber 50% do tráfego (1 instância sobrecarregada). Com Cross-Zone Load Balancing ativado, o tráfego é distribuído uniformemente por todas as instâncias de todas as AZs.

⚠
Cross-Zone: diferença por tipo — ALB tem cross-zone ativado por padrão (sem custo extra). NLB e GLB têm desativado por padrão, e cobram por tráfego inter-AZ quando ativado. No exame, questões de custo de dados podem testar essa diferença.

Connection Draining / Deregistration Delay

Quando o ASG decide terminar uma instância (scale-in, deploy, falha), o ELB para de enviar novas conexões para ela mas aguarda as conexões existentes terminarem — padrão 300 segundos. Esse período é o Deregistration Delay (ALB/NLB) ou Connection Draining (CLB). Ajuste para menos (ex.: 30s) em aplicações stateless, e para mais em aplicações com sessões longas.

Armazenamento — S3, EFS, FSx e Storage Gateway

Prioridade Alta

O SAA-C03 testa profundamente a diferença entre tipos de armazenamento (objeto, bloco, arquivo), as nuances das classes do S3, segurança e controle de acesso a buckets, e quando usar cada serviço de sistema de arquivos. Este capítulo cobre o S3 em profundidade — a maioria das questões de armazenamento envolve o S3.

S3 — conceitos fundamentais

O S3 (Simple Storage Service) é um armazenamento de objetos: arquivos imutáveis endereçados por bucket + chave. Um objeto pode ter até 5 TB; para fazer upload de arquivos maiores que 100 MB, a prática recomendada é o Multipart Upload (paralelo, retomável, obrigatório acima de 5 GB).

  • Durabilidade: 99.999999999% (11 noves) — projetada para sobreviver perda de 2 AZs simultâneas (exceto One Zone-IA)
  • Disponibilidade: varia por classe (Standard = 99.99% SLA)
  • Consistência: strong read-after-write consistency para todos os objetos (desde 2020) — sem "eventual consistency" no S3
  • Namespace global: nomes de bucket são únicos globalmente na AWS; o bucket em si reside numa região específica

S3 Versionamento e proteção contra exclusão

Com o Versionamento ativo, cada versão de um objeto coexiste no bucket. Deletar um objeto insere um "delete marker" em vez de remover permanentemente — para recuperar, delete o delete marker. Para deletar permanentemente uma versão, você precisa especificar o Version ID.

  • MFA Delete: exige MFA para deletar versões ou desativar o versionamento — proteção contra exclusão acidental/maliciosa
  • Object Lock (WORM): impede deleção/modificação por um período. Modos:
    • Governance: usuários com permissão especial podem remover
    • Compliance: ninguém (nem root) pode remover durante o período — para requisitos regulatórios rígidos (SEC, CFTC)

S3 Event Notifications e S3 Batch Operations

S3 Event Notifications disparam ações em resposta a eventos de objetos (PutObject, DeleteObject, etc.) para: Lambda, SQS, SNS ou EventBridge. Caso de uso clássico: processar imagens ao fazer upload → Lambda redimensiona e salva em outro bucket.

S3 Batch Operations executam ações em massa em objetos existentes: copiar, invocar Lambda, restaurar do Glacier, replicar retroativamente. Usa um manifest (lista de objetos ou resultado de S3 Inventory).

S3 Performance — dicas para escala

  • Prefixos paralelos: S3 suporta 3.500 PUT/COPY/POST/DELETE e 5.500 GET/HEAD por prefixo por segundo. Distribuir objetos em prefixos diferentes aumenta o throughput proporcionalmente.
  • S3 Transfer Acceleration: usa os edge locations do CloudFront para upload via backbone da AWS — melhora uploads de clientes geographicamente distantes do bucket.
  • Multipart Upload: obrigatório para objetos acima de 5 GB, recomendado acima de 100 MB. Paralelo, retomável.
  • S3 Select / Glacier Select: executa SQL simples no servidor para buscar apenas parte de um objeto (ex.: filtrar colunas de um CSV de 10 GB) — reduz dados transferidos.

S3 Website Hosting e OAC com CloudFront

O S3 pode hospedar sites estáticos diretamente, mas o padrão arquitetural recomendado é: bucket privado (Block Public Access ativado) + CloudFront com OAC (Origin Access Control). O OAC permite que apenas o CloudFront acesse o bucket via uma política especial, sem tornar o bucket público. Isso adiciona CDN, HTTPS e proteção DDoS automaticamente.

VPC e Redes

Prioridade Alta

Rede é o domínio com mais armadilhas conceituais do exame: a diferença entre stateful/stateless, entre serviços que parecem similares (Peering vs Transit Gateway, VPN vs Direct Connect, CloudFront vs Global Accelerator) é exatamente o que o SAA-C03 testa. Entender a fundo o caminho que um pacote percorre dentro e fora da VPC resolve a maioria das questões.

Anatomia de uma VPC

Uma VPC (Virtual Private Cloud) é uma rede logicamente isolada dentro de uma região AWS, definida por um bloco CIDR (ex.: 10.0.0.0/16). Dentro dela, sub-redes (subnets) são criadas em AZs específicas — uma subnet nunca cruza AZs.

  • Sub-rede pública: tem rota para um Internet Gateway na tabela de rotas.
  • Sub-rede privada: sem rota direta para a internet; usa NAT Gateway/Instance para saída.
  • Route Table: define para onde o tráfego de cada CIDR de destino é enviado. Cada subnet está associada a exatamente uma route table (pode ser a "main" por padrão).
  • 5 IPs reservados por subnet: a AWS reserva o primeiro (rede), segundo (router VPC), terceiro (DNS), quarto (reservado para uso futuro) e o último (broadcast) — um /24 tem 256 IPs, mas só 251 utilizáveis.

Security Group × NACL — o mantra

Esta é a comparação mais cobrada de redes. Memorize as diferenças abaixo — qualquer questão sobre "bloquear um IP específico" ou "regras de retorno" depende disso.

Security Group

Nível de instância (ENI). Stateful — o tráfego de retorno é permitido automaticamente, mesmo sem regra de saída explícita. Só regras de Allow (não existe Deny). Pode referenciar outro SG como origem/destino (ex.: SG-web → SG-app → SG-db), o que elimina a necessidade de saber IPs.

NACL

Nível de sub-rede — afeta todas as instâncias nela. Stateless — é preciso liberar ida E volta explicitamente (ex.: porta efêmera 1024-65535 de volta para respostas HTTP). Permite Allow e Deny, avaliadas em ordem numérica crescente até a primeira regra que casar. É o único mecanismo nativo de VPC que bloqueia um IP específico.

⚠
Regras efêmeras na NACL — Se um cliente externo faz uma requisição HTTP (porta 80) à sua instância, a resposta sai por uma porta efêmera aleatória (1024-65535). A NACL de saída precisa permitir essa faixa, senão a resposta é descartada mesmo que a entrada tenha sido permitida. Esse é um cenário clássico de "o SG está certo mas a NACL está bloqueando o retorno".

VPC Peering em detalhe

Peering cria uma conexão ponto-a-ponto entre duas VPCs (mesma conta ou contas diferentes, mesma região ou inter-região). Pontos de atenção do exame:

  • Não transitivo: se você precisa que A converse com C através de B, precisa de um peering direto A↔C — peering não roteia através de um terceiro.
  • CIDR não pode sobrepor: VPCs com blocos CIDR conflitantes não podem ser peered.
  • Cada lado precisa atualizar sua route table apontando para o peering connection.

Transit Gateway — o hub que substitui a teia de peerings

Quando o número de VPCs cresce, o número de peerings necessários para conectar todas entre si cresce como N×(N-1)/2 — rapidamente inviável. O Transit Gateway resolve isso atuando como um roteador regional central: cada VPC/VPN/Direct Connect se conecta uma vez ao TGW, e o TGW roteia entre todos. Suporta políticas de roteamento e segmentação entre grupos de VPCs (route tables separadas no TGW).

Health checks monitoram endpoints (HTTP/HTTPS/TCP) ou outros health checks (calculados) ou alarmes do CloudWatch. São a base do Failover Routing — sem health check, o Route 53 não sabe quando trocar para o secundário.

CloudFront × Global Accelerator — não cache vs cache

CloudFront (CDN)

Faz cache de conteúdo nos edge locations. HTTP/HTTPS apenas. Casos: "conteúdo estático global", "streaming de vídeo", "reduzir carga na origem", "proteção DDoS na borda" (integra com Shield/WAF). OAC (Origin Access Control) protege um S3 privado, servindo apenas via CloudFront.

Global Accelerator

Sem cache — é um atalho de rede que usa o backbone privado da AWS em vez da internet pública. Suporta TCP e UDP. Casos: "gaming", "VoIP", "IoT", "IP estático global (2 IPs anycast fixos)", "failover entre regiões em segundos" (muito mais rápido que TTL de DNS).

💡
Quando os dois aparecem juntos na mesma questão: se o conteúdo é cacheável (HTML, imagens, vídeo) → CloudFront. Se o protocolo é não-HTTP (jogos, voz) ou a aplicação precisa de IP fixo + menor latência de rede sem cache → Global Accelerator. Os dois não competem — atacam problemas diferentes.

VPC Flow Logs e Network Firewall

VPC Flow Logs capturam metadados de tráfego IP (origem, destino, porta, protocolo, ACCEPT/REJECT) entrando e saindo de ENIs, subnets ou VPCs inteiras — vão para CloudWatch Logs ou S3. Use para auditoria de segurança, troubleshooting de conectividade ("por que esse tráfego está sendo bloqueado?"), e detecção de tráfego anômalo.

AWS Network Firewall é um firewall stateful gerenciado para a VPC inteira — inspeciona até a camada de aplicação (deep packet inspection), suporta regras Suricata-compatible. Use quando SG/NACL não bastam (ex.: bloquear domínios específicos, inspecionar payload).

Checkpoint de revisão

  • Global Accelerator — suporta TCP/UDP, fornece IPs anycast estáticos, e usa o backbone privado da AWS em vez da internet pública para menor latência.
  • NACL com regra Deny para o IP de origem. Security Group não suporta regra de Deny explícito — só permite ou simplesmente não inclui a regra (negação implícita).
  • Transit Gateway — hub centralizado que evita criar N×(N-1)/2 conexões de peering. Cada VPC e a conexão on-premises (via VPN ou Direct Connect) se conectam uma vez ao TGW.

  • AWS Systems Manager Session Manager — usa o agente SSM instalado na instância e credenciais IAM para estabelecer a sessão, sem precisar de porta 22 aberta nem bastion host. Todo acesso é auditável via CloudTrail.

RDS — fundamentos operacionais

  • Backups automatizados: snapshot diário + transaction logs, retenção 0-35 dias, suporta restauração point-in-time dentro da janela.
  • Snapshots manuais: retidos indefinidamente até serem deletados manualmente — independentes da retenção automática.
  • Parameter Groups: configuram parâmetros do motor (ex.: max_connections). Algumas mudanças exigem reboot.
  • Option Groups: habilitam recursos específicos do motor (ex.: Oracle TDE, SQL Server Mirroring).
  • Encryption at rest: deve ser definida na criação — não pode ser ativada depois num banco já existente sem snapshot+restore com encryption habilitada.
  • Maintenance Window: janela para patches obrigatórios do motor — em Multi-AZ, o patch acontece primeiro no standby, depois faz failover.

RDS Proxy — pool de conexões gerenciado

Quando Lambda ou aplicações com muitas conexões efêmeras acessam o RDS/Aurora diretamente, cada invocação pode abrir uma nova conexão — esgotando o limite de conexões do banco rapidamente. O RDS Proxy fica entre a aplicação e o banco, fazendo pooling e multiplexação de conexões, reduzindo drasticamente o número de conexões reais no banco. Também acelera failover (mantém conexões do proxy enquanto o banco reconecta).

DynamoDB — modelo de dados e capacidade

DynamoDB é NoSQL chave-valor/documento totalmente gerenciado, com latência de milissegundos de forma consistente independente da escala. Cada item é identificado por uma Partition Key (PK) e opcionalmente uma Sort Key (SK) — juntas formam a chave primária composta.

⚠
Armadilha: Query vs Scan — Query usa a chave de partição (e opcionalmente a de ordenação) para buscar diretamente — rápido, previsível e barato. Scan percorre a tabela inteira item por item — lento e caro em tabelas grandes, mesmo com filtro (o filtro é aplicado depois de ler todos os itens). Se a questão menciona performance ruim em tabela grande, a resposta envolve modelar a chave de partição corretamente ou criar um GSI — nunca "adicionar um FilterExpression no Scan".

Hot Partition — o problema de modelagem mais cobrado

O DynamoDB distribui dados e capacidade entre partições físicas baseado na partition key. Se muitos itens compartilham a mesma PK (ex.: status="ATIVO" para milhões de registros) ou se um pequeno conjunto de chaves recebe tráfego desproporcional, essa partição fica sobrecarregada (hot partition) mesmo com capacidade total disponível na tabela.

💡
Solução padrão: redesenhar a PK para ter alta cardinalidade (ex.: incluir um sufixo aleatório ou o ID do usuário na chave) distribuindo a carga entre mais partições. "Aumentar capacidade provisionada" não resolve hot partition — o problema é distribuição, não capacidade total.

ElastiCache — Redis × Memcached

Redis

Estruturas ricas (listas, sets, sorted sets, hashes). Persistência (snapshots + AOF) + replicação + Multi-AZ com failover automático. "Ranking/leaderboard" = sorted sets. Suporta Pub/sub, scripts Lua, e Cluster Mode para sharding horizontal.

Memcached

Cache simples chave-valor, multi-thread (usa múltiplos cores por nó). Sem persistência nem replicação — dado é puramente descartável/recriável. Escala horizontal simples adicionando nós (sharding automático no client).

Padrões de cache mais cobrados: Lazy Loading (busca no cache; se não tem, busca no banco e popula o cache — simples, mas a primeira leitura é sempre um cache miss) e Write-Through (toda escrita no banco também escreve no cache — cache sempre atualizado, mas escreve dados que talvez nunca sejam lidos).

Redshift — data warehouse colunar

Redshift é otimizado para OLAP — queries analíticas complexas sobre grandes volumes, não para transações OLTP de alto volume. Armazenamento colunar + compressão agressiva + distribuição entre nós. Redshift Spectrum permite consultar dados direto no S3 sem carregá-los no cluster. Redshift Serverless elimina a gestão de cluster, escalando automaticamente.

Alta disponibilidade e escalabilidade

Prioridade Alta

EC2, ASG e ELB (Cap. 2) dão os blocos de construção. Este capítulo foca em como combiná-los em padrões de arquitetura resilientes, e na camada de mensageria que desacopla componentes — o ingrediente que separa uma arquitetura "que funciona" de uma que "aguenta picos e falhas parciais sem cair".

HA × Tolerância a falhas × Resiliência — vocabulário que muda a resposta

💡
Definições precisas que o exame usa:
Alta disponibilidade (HA): minimizar downtime — redundância N+1 em AZs diferentes atrás de um balanceador, com failover que leva segundos a minutos.
Tolerância a falhas (FT): operar SEM degradação perceptível durante a falha — exige redundância ativa duplicada processando em paralelo o tempo todo. Custa significativamente mais que HA.
Resiliência: capacidade do sistema de se recuperar de falhas e continuar operando — engloba HA, FT, backups e DR como ferramentas.
Se o enunciado diz "sem impacto perceptível ao usuário durante a falha", é FT. Se aceita "breve interrupção até o failover", é HA.

Retry com Exponential Backoff e Jitter

Quando uma chamada falha (throttling, erro transitório), repetir imediatamente piora o problema — gera uma onda sincronizada de retries que sobrecarrega ainda mais o serviço (retry storm). A prática recomendada:

Backoff exponencial com jitter (aleatoriedade):
Tentativa 1: falha → espera ~1s
Tentativa 2: falha → espera ~2s (+ jitter aleatório)
Tentativa 3: falha → espera ~4s (+ jitter aleatório)
Tentativa 4: falha → espera ~8s (+ jitter aleatório)
O jitter evita que múltiplos clientes re-tentem exatamente no mesmo instante

Os SDKs da AWS já implementam isso nativamente nas chamadas de API — é por isso que throttling ocasional do KMS/DynamoDB raramente quebra uma aplicação bem configurada.

Um tópico SNS pode ter múltiplos subscribers, mas nem todos precisam receber toda mensagem. Subscription Filter Policies permitem que cada subscriber só receba mensagens cujos atributos casem com seu filtro — evitando que cada consumidor precise filtrar no próprio código (e gastar invocações de Lambda com mensagens irrelevantes).

Checkpoint de revisão

  • SQS FIFO — garante exactly-once processing e ordem dentro do mesmo Message Group ID. SQS Standard só garante "melhor esforço" de ordem e pode entregar duplicatas (at-least-once).
  • SNS fan-out para 3 filas SQS, cada uma com seu consumidor (Lambda ou worker). Cada sistema processa de forma independente, no seu próprio ritmo, sem que a falha de um afete os outros.
  • Scheduled Scaling — escala antes das 9h, eliminando o lag do scaling reativo. Predictive Scaling é uma alternativa que detecta o padrão automaticamente via ML, sem precisar configurar o horário manualmente.
  • HA aceita um pequeno downtime durante o failover (segundos a minutos). FT exige redundância ativa duplicada operando o tempo todo, sem qualquer degradação perceptível — custa significativamente mais e só é exigido quando o enunciado especifica "zero impacto" ou "sem degradação perceptível".
  • Circuit Breaker — para de chamar o serviço problemático após detectar falhas repetidas, retornando erro/fallback imediatamente em vez de esperar timeout a cada chamada. Depois de um período, testa novamente (half-open) antes de voltar ao normal.
  • Implementar exponential backoff com jitter: aumentar o tempo de espera exponencialmente entre tentativas, com um componente aleatório (jitter) para evitar que múltiplos clientes re-tentem no mesmo instante exato (retry storm). Os SDKs da AWS já fazem isso nativamente.

Disaster Recovery

Prioridade Alta

RPO e RTO — os dois números que definem tudo

          ←— RPO —→     🔥     ←———— RTO ————→
    último backup    desastre    sistema no ar de novo
    (dados perdidos)            (tempo parado)
💡 Definições RPO (Recovery Point Objective) = quanto de dado posso perder? Olha para o passado — depende de quando foi o último backup/replicação. RTO (Recovery Time Objective) = quanto tempo posso ficar fora do ar? Olha para o futuro — depende de quanto tempo leva para reestabelecer o serviço. Quanto menores os dois, mais cara a estratégia.

As 4 estratégias — da mais barata à mais cara

Warm Standby: a aplicação JÁ ESTÁ rodando (mini-versão), mas em capacidade reduzida. Poderia atender uma requisição agora mesmo.

AWS Backup — backups centralizados e governança

O AWS Backup centraliza e automatiza backups de EC2, EBS, RDS, Aurora, DynamoDB, EFS, FSx, Storage Gateway e mais — em vez de configurar backup serviço por serviço.

  • Backup Plans: definem frequência, janela, retenção e regras de transição para cold storage (mais barato após N dias).
  • Backup Vault Lock: modo WORM para o vault de backups — impede até o administrador de deletar backups antes do prazo, proteção contra ransomware e exclusão maliciosa/acidental.
  • Cross-Region Copy e Cross-Account Copy: copia backups para outra região/conta automaticamente — essencial para DR de região e isolamento contra comprometimento de conta.
  • Backup Audit Manager: avalia se os planos de backup atendem requisitos de compliance definidos.

AWS Elastic Disaster Recovery (DRS) em detalhe

O DRS (sucessor do CloudEndure) instala um agente leve nos servidores de origem (on-premises, outra cloud, ou EC2) que replica continuamente as alterações de disco bloco a bloco para uma área de staging de baixo custo na AWS. Durante um desastre, o DRS lança instâncias EC2 completas a partir dessa réplica em minutos — diferente de um backup tradicional, que exigiria restaurar e reconfigurar do zero.

AWS Resilience Hub — medindo se sua arquitetura atinge o RPO/RTO desejado

O Resilience Hub avalia uma aplicação (definida via CloudFormation, Terraform, ou descoberta de recursos) contra metas de RPO/RTO definidas, simulando cenários de falha (AZ down, região down, falha de componente) e apontando gaps específicos — ex.: "este RDS não tem Multi-AZ, o que viola o RTO de 5 min definido". Gera recomendações de remediação concretas.

Testando DR — Chaos Engineering e Game Days

Uma estratégia de DR documentada mas nunca testada é uma suposição, não uma garantia. O AWS Fault Injection Service (FIS) permite injetar falhas controladas em produção (terminar instâncias, simular latência de rede, esgotar CPU) para validar que mecanismos de auto-healing e failover realmente funcionam como esperado — a prática conhecida como Chaos Engineering. "Game Days" são exercícios programados onde o time simula um desastre real e segue o runbook de DR para medir o RTO de verdade.

Checkpoint de revisão

  • Pilot Light. Dados replicando continuamente (garante RPO de minutos) + infra desligada (barata). RTO de 1-2h para "acender" a infra é viável.
  • Multi-site Active/Active. Duas ou mais regiões em produção plena, com Route 53 ou Global Accelerator distribuindo o tráfego.
  • Warm Standby. A aplicação já está rodando na DR (distingue de Pilot Light onde ela está desligada).
  • Se toda a região está indisponível, qualquer recurso armazenado nela — incluindo backups e snapshots — também fica inacessível. DR de região exige cópia em outra região (CRR, AWS Backup com cópia cross-region, ou réplicas cross-region).
  • AWS Backup tira snapshots periódicos (RPO em horas). DRS replica continuamente bloco a bloco para a região de DR, permitindo RPO de segundos e RTO de minutos — é o sucessor do CloudEndure para lift-and-shift de servidores inteiros.
  • AWS Backup Vault Lock em modo Compliance — torna os backups imutáveis por um período, impedindo exclusão mesmo por um administrador ou root comprometido. Combine com Cross-Account Copy para isolar backups numa conta separada com acesso restrito.
  • AWS Resilience Hub para avaliar a arquitetura contra as metas de RPO/RTO definidas, e AWS Fault Injection Service (FIS) para injetar falhas controladas (Chaos Engineering) e medir o comportamento real do failover em um "Game Day".

Computação serverless (Lambda, containers)

Prioridade Alta

"Serverless" no exame é sinônimo de menor esforço operacional. Quando o enunciado usa expressões como "least operational overhead", "no servers to manage" ou "without provisioning infrastructure", o leque de respostas plausíveis se reduz drasticamente — geralmente para Lambda, Fargate ou um serviço totalmente gerenciado.

Lambda SnapStart — eliminando cold start sem custo extra

Para runtimes como Java (que historicamente tem cold starts lentos), o SnapStart tira um snapshot do ambiente de execução já inicializado (após o init code rodar) e o restaura em invocações subsequentes — reduzindo cold start de segundos para milissegundos, sem o custo extra do Provisioned Concurrency. Diferente do Provisioned Concurrency, que mantém ambientes "quentes" o tempo todo (e cobra por isso), o SnapStart restaura de um snapshot sob demanda.

Lambda Layers e Function URLs

Layers empacotam dependências/bibliotecas compartilhadas entre múltiplas funções, separadas do código principal — reduz duplicação e acelera deploys (o código muda, a layer com dependências pesadas não). Function URLs dão a uma Lambda um endpoint HTTPS dedicado sem precisar de API Gateway — mais simples para casos de uso básicos (webhook único), mas sem os recursos de gerenciamento de API (throttling avançado, API keys, usage plans) que o API Gateway oferece.

Prioridade Alta

KMS — o cofre de chaves

O KMS (Key Management Service) gerencia chaves criptográficas usadas para criptografar dados em praticamente todos os serviços AWS. Chaves simétricas (padrão, usadas para a maioria dos casos) e assimétricas (par público/privado, para casos específicos de assinatura ou criptografia entre sistemas externos).

Tipo de chaveQuem controlaRotaçãoAuditávelCusto
AWS-ownedAWS (invisível para você)AWS decideNãoGrátis
AWS-managedAWS, mas visível na sua conta (ex.: aws/s3)Automática anualSim — uso no CloudTrailGrátis

Chaves do KMS são regionais — não saem da região onde foram criadas (exceto Multi-Region Keys, que replicam a mesma chave para outras regiões mantendo o mesmo Key ID, úteis para DR ou Aurora Global Database com criptografia).

Envelope Encryption — como o KMS realmente criptografa grandes volumes

O KMS não criptografa diretamente arquivos grandes (limite de 4 KB por chamada). Em vez disso, usa envelope encryption: o KMS gera uma Data Key, que criptografa o dado localmente (no SDK do cliente ou no serviço AWS), e a própria Data Key é então criptografada pela CMK e armazenada junto com o dado. Para descriptografar, primeiro descriptografa a Data Key via KMS, depois usa-a para descriptografar o dado. Isso é o que permite ao S3/EBS/RDS criptografar objetos de qualquer tamanho usando o KMS por trás.

⚠
Armadilha de snapshot/cross-account criptografado — Compartilhar um snapshot EBS/RDS criptografado com outra conta exige compartilhar acesso à CMK também (via key policy) — sem isso, a outra conta recebe o snapshot mas não consegue descriptografá-lo, ficando inutilizável. Chaves AWS-managed não podem ser compartilhadas entre contas — só CMK.

CloudHSM — quando KMS não é suficiente

O CloudHSM oferece hardware de criptografia dedicado (single-tenant) dentro de uma VPC, atendendo padrões de compliance que exigem controle exclusivo do hardware (FIPS 140-2 Level 3). Diferente do KMS (multi-tenant, gerenciado), você é responsável por gerenciar o cluster HSM. Use apenas quando o enunciado exige explicitamente esse nível de isolamento — para a maioria dos casos, KMS é a resposta mais simples e suficiente.

Secrets Manager × Parameter Store

Secrets Manager

Rotação automática nativa (RDS, Redshift, DocumentDB têm integração pronta; outros via Lambda customizada). Pago por segredo/mês + por API call. "Rotacionar senha do banco automaticamente" = Secrets Manager. Suporta replicação cross-region nativa.

Parameter Store (SSM)

Tier Standard grátis (Advanced tem custo). Guarda configs e segredos com criptografia KMS opcional. Sem rotação nativa — exigiria automação própria. "Guardar configuração/segredo de forma barata, sem rotação automática" = Parameter Store.

Encryption in Transit — TLS e certificados

Encryption at rest (KMS) protege dados parados. Encryption in transit protege dados em movimento via TLS/HTTPS — igualmente cobrado no exame.

X-Ray — rastreando requisições distribuídas

Quando o enunciado descreve uma arquitetura com múltiplos microsserviços (API Gateway → Lambda → DynamoDB → SNS, por exemplo) e pede para identificar onde está o gargalo de latência ou um erro em cadeia, a resposta é AWS X-Ray — ele traça o caminho completo de uma requisição (trace) através dos serviços e mostra um mapa de serviço com tempos de cada salto.

Para reduzir custo/volume, o X-Ray usa sampling rules — não traceia 100% das requisições por padrão (ex.: 1 req/s + 5% das demais). Annotations (indexáveis, usadas em filtros) e Metadata (não indexável, só para contexto) são as duas formas de anexar dados extras a um trace.

💡
Sinal prático — "Identificar qual componente está causando latência numa cadeia de microsserviços" → X-Ray. "O que aconteceu com a CPU/memória/disco?" → CloudWatch. São perguntas diferentes — não confunda observabilidade de uma requisição (X-Ray, tracing) com métricas de infraestrutura (CloudWatch, métricas agregadas).

Analytics (Athena, Glue, Kinesis, Redshift)

Base

O Domínio 3 (alta performance) cobra analytics principalmente como "qual serviço processa/consulta este tipo e volume de dados com menos esforço operacional". A tabela abaixo cobre praticamente todo o vocabulário que aparece nessas questões.

ServiçoUma linha para o SAA
AthenaSQL serverless direto no S3, paga por dado escaneado. Otimize com Parquet + particionamento
GlueETL serverless + Data Catalog (catálogo de metadados usado por Athena/Redshift Spectrum/EMR)
Kinesis Data StreamsIngestão streaming em tempo real, ordenada, com replay. Múltiplos consumidores
Data FirehoseEntrega streaming near-real-time direto p/ S3/Redshift, zero administração
RedshiftData warehouse para BI/OLAP em escala — não para OLTP

Pipeline de streaming — Kinesis Data Streams × Data Firehose

Kinesis Data Streams

Você controla os consumidores (shards, retenção até 365 dias, replay de eventos). Vários consumidores independentes podem ler o mesmo stream. Use quando a aplicação precisa processar os eventos em tempo real com lógica própria.

Data Firehose

Totalmente gerenciado — sem shards para administrar. Entrega automaticamente para S3, Redshift, OpenSearch, com transformação opcional via Lambda. Use quando o destino é só "carregar os dados em algum lugar" sem lógica de consumo customizada.

⚠️ Athena vs Redshift — a distinção favorita "Analisar logs do ALB/VPC Flow que já estão no S3, com SQL, menor custo e esforço" → Athena. Não Redshift, não EMR. Athena é serverless e consulta S3 diretamente sem carregar dado.
💡 Sinal prático "Reduzir o custo de queries no Athena" quase sempre tem a mesma resposta: converter os dados para um formato colunar comprimido (Parquet ou ORC) e particionar por data/categoria — Athena cobra por bytes escaneados, então menos dados lidos = menos custo.

Quando escolher EMR em vez de Glue

Glue é a resposta padrão para ETL serverless simples. EMR entra quando o enunciado menciona frameworks específicos (Spark, Hive, Presto, HBase) com necessidade de configuração detalhada do cluster, jobs de longa duração com estado complexo, ou quando o time já tem expertise em Hadoop/Spark e precisa de controle total sobre o cluster — incluindo o uso de instâncias Spot para os nós de tarefa, reduzindo custo de processamento em até 90%.

Lake Formation — governança sobre o data lake

Construir um data lake no S3 com Glue Catalog resolve o armazenamento, mas não o controle de acesso granular. Lake Formation adiciona permissões em nível de tabela, coluna e até linha (row-level security) sobre os dados catalogados, com um único lugar para conceder acesso a múltiplos serviços (Athena, Redshift Spectrum, EMR) — em vez de replicar políticas de bucket S3 individualmente para cada consumidor.

QuickSight — visualização e BI

QuickSight é o serviço de BI serverless da AWS — conecta a Athena, Redshift, RDS, S3 e outras fontes para criar dashboards interativos. Usa o motor SPICE (in-memory) para acelerar consultas repetidas sem golpear a fonte de dados a cada visualização.

Redshift Spectrum — consultando o S3 sem carregar dados

O Redshift Spectrum estende as queries SQL do Redshift para consultar dados direto no S3 (via Glue Catalog), sem precisar fazer ETL para carregar tudo no cluster primeiro. Útil para juntar dados "quentes" (já no Redshift) com dados "frios" históricos (que ficam mais baratos no S3) numa única query.

OpenSearch — busca e análise de logs

OpenSearch (fork open-source do Elasticsearch mantido pela AWS) é a resposta quando o enunciado pede busca full-text, análise de logs em tempo quase real, ou dashboards de observabilidade (via OpenSearch Dashboards). Diferente do Athena (SQL sobre arquivos estáticos no S3), OpenSearch é um motor de busca/indexação com ingestão contínua — mais adequado para "buscar por palavra-chave em milhões de logs" do que Athena.

  • EMR (Elastic MapReduce) — Hadoop/Spark gerenciado com controle total da configuração do cluster, suportando instâncias Spot nos task nodes para reduzir custo.

O capítulo 8 já apresentou containers como um degrau da "escada de compute". Aqui vamos abrir o ECS e o EKS por dentro — como cada peça se encaixa, quando Fargate é a escolha óbvia, e os detalhes operacionais que o exame adora cobrar (modos de rede, capacity providers, estratégias de deploy).

ECS: as quatro peças que formam um serviço

TaskUma instância em execução da task definition — pode ser standalone (batch) ou parte de um serviço

Capacity providers: onde as tasks rodam

OpçãoQuando usar
FargateSem servidor para gerenciar — paga por vCPU/memória da task. Ótimo para cargas variáveis e times pequenos
Fargate SpotAté 70% mais barato, mas a task pode ser interrompida com 2 min de aviso — ideal para workers/batch tolerantes a falha

EKS: Kubernetes gerenciado

O EKS gerencia o control plane (API server, etcd) com SLA da AWS. Você ainda escolhe onde rodam os pods:

Managed Node Groups

EC2 (On-Demand ou Spot) provisionadas e atualizadas pelo EKS via Auto Scaling Groups. Você escolhe instance type/AMI. Melhor custo em escala e acesso a recursos de nó (DaemonSets, GPU, volumes EBS).

Fargate Profiles

Cada pod roda isolado em sua própria "mini-VM" Fargate — sem gerenciar nós. Não suporta DaemonSets, hostPort ou volumes EBS diretos. Bom para workloads simples e multi-tenant.

⚠️ Armadilha: modo de rede awsvpc e limite de ENIs No modo awsvpc (obrigatório no Fargate, recomendado no EC2), cada task recebe sua própria ENI com IP da VPC — isso significa que o número de tasks por instância EC2 fica limitado pelo número de ENIs que o tipo de instância suporta (ENI trunking ajuda, mas é mais um limite a verificar). Em subnets pequenas, muitas tasks Fargate podem esgotar os IPs disponíveis — dimensione a subnet pensando em containers, não só em instâncias EC2.

Quando microserviços em containers precisam se encontrar dinamicamente (sem IPs fixos), o ECS Service Connect (ou App Mesh / AWS Cloud Map para casos mais customizados) fornece nomes DNS internos estáveis que resolvem para as tasks saudáveis atuais, com métricas de tráfego e retry automático entre serviços.

O padrão Sidecar roda um container auxiliar na mesma task que o container principal, compartilhando rede/storage — usado para proxies (Envoy no App Mesh), agentes de log (Fluent Bit enviando para CloudWatch Logs) ou coleta de métricas (ADOT collector), sem acoplar essa responsabilidade ao código da aplicação.

API Gateway, Step Functions e orquestração serverless

Consolidar

O capítulo 8 cobriu Lambda a fundo, mas tratou API Gateway e orquestração só de passagem. Aqui entramos nos detalhes que separam "funciona no lab" de "passa na prova": tipos de API, controle de tráfego, autenticação, e como coordenar múltiplas Lambdas sem amarrar tudo em código.

REST API vs HTTP API vs WebSocket API

TipoQuando usar
HTTP APIPadrão para APIs REST modernas — até ~70% mais barata e com menor latência que REST API. Suporta JWT authorizers e integração direta com Lambda/HTTP backends
REST APIRecursos completos: API keys, usage plans, transformação de request/response, validação de schema, caching nativo, integração com WAF e endpoints privados via VPC endpoint

💡 Regra rápida Se o enunciado não menciona API keys, usage plans, caching ou WAF — e só quer "expor uma Lambda via HTTP da forma mais barata/rápida" — a resposta é HTTP API. Se menciona "monetizar por chave de API" ou "transformar o payload antes de chegar no backend", é REST API.

Controle de acesso e de tráfego

MecanismoPara quê
Throttling (rate/burst)Limite padrão de conta (token bucket) — protege os backends de picos. Pode ser configurado por método ou por usage plan

Lambda AuthorizerLógica de autorização customizada (valida token JWT/OAuth de terceiros, chama serviço externo)

⚠️ Armadilha: 29 segundos vs 15 minutos O API Gateway tem um timeout fixo de 29 segundos — não configurável — para qualquer integração, mesmo que a Lambda por trás permita até 15 minutos. Se o processamento pode demorar mais que isso, a API Gateway não pode chamar a Lambda de forma síncrona e esperar o resultado. Padrão correto: a API retorna 202 imediatamente (ex.: enfileira em SQS ou inicia uma Step Functions), e o cliente consulta o resultado depois (polling) ou recebe via WebSocket/SNS.

Stages, Caching e Canary no API Gateway

Cada Stage (dev, staging, prod) é uma versão deployada independente da API, com suas próprias variáveis de configuração e logs. Caching nativo do REST API armazena respostas por TTL configurável, reduzindo chamadas ao backend para dados que não mudam a cada requisição. Canary Release permite direcionar uma % do tráfego para uma nova versão da API antes de promovê-la 100% — reduzindo o risco de um deploy ruim afetar todos os usuários de uma vez.

Step Functions: Standard vs Express

Standard Workflows

Duração até 1 ano, execução exactly-once, histórico de execução visual completo, preço por transição de estado. Ideal para workflows de longa duração com aprovações humanas — processos de negócio, orquestração de pipelines.

Express Workflows

Duração até 5 minutos, execução at-least-once, preço por número de execuções + duração/memória (como Lambda). Ideal para alto volume de eventos — processamento de stream, ingestão de dados — com histórico detalhado no CloudWatch Logs.

Agendamento de tarefas

NecessidadeServiço
"Rodar esta Lambda toda noite às 2h"EventBridge Scheduler — expressões cron/rate, fuso horário, alvo flexível (Lambda, Step Functions, SQS, ECS RunTask, APIs de terceiros via conector)
"Reagir quando X acontecer no meu ambiente AWS" (ex.: objeto criado no S3, instância EC2 muda de estado)EventBridge Rules — barramento de eventos com filtro por padrão (event pattern matching)
Milhares de schedules independentes por cliente/tenantEventBridge Scheduler — suporta até milhões de schedules, criados/gerenciados dinamicamente via API

Padrão de orquestração serverless

// Cliente chama API Gateway → inicia Step Functions (assíncrono, evita o timeout de 29s)
Cliente → API Gateway → StartExecution("PipelineProcessamento")
                        ↓
              Step Functions (Standard)
                ├─ Lambda: valida entrada
                ├─ Lambda: chama serviço de IA (cap. 18)
                ├─ Choice: sucesso? → próximo : Catch → notifica SNS
                └─ Lambda: grava resultado no DynamoDB
★ Quando Step Functions substitui "Lambda chamando Lambda" Se você está tentado a fazer uma Lambda invocar outra Lambda de forma síncrona (e mais outra, e mais outra), pare: isso cria acoplamento forte, dificulta debug e pode gerar custo duplicado de espera. Step Functions move a lógica de "o que vem depois" para fora do código — cada Lambda fica pequena e independente, com retries, timeouts e tratamento de erro configurados visualmente no workflow.

  3. 3

  4. 4

Checkpoint de revisão

  • HTTP API — mais barata e com menor latência que REST API, suficiente quando não há necessidade dos recursos avançados (usage plans, caching, transformação de payload).
  • Standard Workflow — suporta execuções de até 1 ano, com estados de espera (ex.: "Wait for Task Token") até alguém aprovar manualmente.
  • Express Workflow — preço por execução/duração compensa em alto volume, e a semântica at-least-once é aceitável para esse tipo de processamento.
  • O API Gateway tem timeout fixo de 29s, independente do timeout da Lambda (até 15 min). Corrigir tornando a chamada assíncrona: a API retorna 202 + um id de job (ex.: inicia Step Functions ou enfileira em SQS), e o cliente consulta o status depois.
  • REST API com endpoint do tipo Private, associado a um VPC endpoint (Interface Endpoint) — o tráfego nunca sai para a internet pública, e uma resource policy controla quais VPCs/contas podem chamá-la.

Tipos de registro: Alias vs CNAME

RegistroCaracterística
Alias (A/AAAA)Extensão da AWS — aponta para recursos AWS (ALB, CloudFront, S3 website, outro registro Route 53) de graça (sem cobrança de consulta) e funciona no apex da zona (ex.: exemplo.com, sem subdomínio)

Health checks do Route 53

TipoComo funciona
Basic (endpoint)Verifica um IP/domínio diretamente — HTTP/HTTPS/TCP — com checadores globais distribuídos avaliando a % de respostas saudáveis

CloudWatch AlarmReflete o estado de um alarme do CloudWatch — útil para monitorar recursos que o Route 53 não alcança diretamente (ex.: RDS, métricas de aplicação)
★ Failover automático = Health Check + Routing Policy Failover Health checks por si só não redirecionam tráfego — eles apenas marcam um registro como saudável/não saudável. É a política de roteamento Failover (cap. 4) que usa esse status para decidir entre o registro primário e o secundário. Sempre que o enunciado pedir "DNS muda automaticamente se a região cair", são as duas peças juntas.

CloudFront: cache e invalidação

ConceitoO que controla

TTL (min/default/max)Quanto tempo o objeto fica no edge antes de revalidar com a origem — pode ser sobrescrito pelos headers Cache-Control/Expires da origem

Acesso privado ao S3: OAC vs OAI

OAC (Origin Access Control) — atual

Suporta SSE-KMS no bucket, todos os métodos HTTP (incluindo POST/PUT para uploads via CloudFront) e assinatura SigV4 completa. É a opção recomendada pela AWS para qualquer distribuição nova.

OAI (Origin Access Identity) — legado

Ainda funciona e aparece em ambientes antigos/questões de prova, mas não suporta SSE-KMS nem todos os métodos. Se a questão menciona bucket criptografado com KMS, OAI não atende — precisa de OAC.

Conteúdo restrito: Signed URLs vs Signed Cookies

CenárioEscolha
Acesso a um único arquivo (ex.: link de download de um PDF)Signed URL — query string com assinatura e expiração, válida para aquele objeto/path
Acesso a múltiplos arquivos (ex.: todos os vídeos de um curso após login)Signed Cookies — o navegador envia o cookie em toda requisição subsequente, sem precisar assinar cada URL
Restringir por geografia/paísGeo Restriction da distribuição (allowlist/denylist por país) — não precisa de signed URL/cookie

Personalização no edge: Lambda@Edge vs CloudFront Functions

CloudFront Functions

JavaScript leve, execução em microssegundos, milhões de requisições/segundo, roda apenas em viewer request/response. Ideal para manipulações simples: reescrever URLs, normalizar headers, checagens de autorização leves.

Lambda@Edge

Funções Lambda (Node.js/Python) com mais recursos e tempo de execução, podem rodar nas 4 fases (viewer/origin request/response) e acessar rede externa. Use quando precisar de lógica mais pesada — ex.: chamar uma API externa antes de servir o conteúdo.

Checkpoint de revisão

  • CNAME não pode coexistir com os registros NS/SOA obrigatórios no apex da zona. A solução é um registro Alias (A/AAAA), que é uma extensão da AWS sem essa restrição e sem custo de consulta.

  • Signed Cookies — um conjunto de cookies assinados é emitido após o login e enviado automaticamente pelo navegador em todas as requisições subsequentes para qualquer objeto autorizado, sem precisar gerar uma URL assinada por arquivo.
  • CloudFront Functions — para validações leves de headers/URLs em viewer request, com latência de microssegundos e custo menor. Lambda@Edge seria over-engineering para essa lógica simples.

Redes avançadas e conectividade híbrida

Base

O capítulo 4 introduziu Transit Gateway, VPN e Direct Connect na tabela de "conectividade". Aqui aprofundamos cada um — especialmente os cenários de múltiplas VPCs/contas/regiões e os detalhes de redundância que o exame cobra em questões de "arquitetura corporativa".

Transit Gateway em profundidade

O TGW é um "roteador regional" central: cada VPC/VPN/Direct Connect se conecta a ele via um attachment, e route tables do TGW (não as route tables da VPC) decidem quem enxerga quem.

RecursoDetalhe
TGW Route TablesCada attachment é associado a uma route table do TGW — permite segmentar (ex.: VPCs de produção não veem VPCs de dev, mesmo conectadas ao mesmo TGW)
Peering entre TGWs (inter-region)Conecta TGWs de regiões diferentes — tráfego permanece na rede backbone da AWS, criptografado
MulticastTGW suporta grupos multicast entre subnets de VPCs anexadas — caso de uso de nicho, mas que aparece como "diferencial" do TGW
Compartilhamento via RAMUm TGW criado na conta de rede central pode ser compartilhado com outras contas da Organization
⚠️ Armadilha: VPC Peering não é transitivo, Transit Gateway é Se a VPC A está peered com B, e B está peered com C, A não consegue alcançar C através de B — cada peering é um túnel ponto a ponto isolado. Para "dezenas de VPCs precisam se comunicar entre si" sem criar uma malha de N×(N-1)/2 peerings, a resposta é Transit Gateway (hub-and-spoke), que roteia entre todos os attachments conforme as route tables do TGW permitirem.

Direct Connect: tipos de interface virtual (VIF)

TipoPara quê
Private VIFAcessa recursos dentro de uma VPC (via Virtual Private Gateway ou Direct Connect Gateway) — IPs privados, como se estivesse na mesma rede
Public VIFAcessa serviços públicos AWS (S3, DynamoDB, etc.) usando seus endpoints públicos, mas pela conexão dedicada — sem passar pela internet
Transit VIFConecta a um Direct Connect Gateway associado a um ou mais Transit Gateways — para alcançar múltiplas VPCs/regiões por uma única conexão física
LAG (Link Aggregation Group)Agrupa múltiplas conexões DX físicas em uma só conexão lógica — mais throughput e redundância física
Direct Connect GatewayPermite que uma conexão DX alcance VPCs em múltiplas regiões (exceto China) através de Private/Transit VIFs

VPN: redundância e performance

ConceitoDetalhe
2 túneis por Site-to-Site VPNCada conexão VPN vem com 2 túneis IPsec para redundância — o ideal é configurar o roteador on-premises para usar ambos (ECMP ou BGP com prioridades)
BGP (dinâmico) vs estáticoBGP permite failover automático de rotas entre os túneis e detecta o caminho mais curto. Roteamento estático exige atualização manual das rotas
VPN acelerada (Global Accelerator)Usa a rede global da AWS para reduzir latência até o endpoint VPN — útil quando o caminho de internet até a região é ruim
VPN como backup do Direct ConnectPadrão comum: DX como link primário (baixa latência, alta largura de banda) + VPN Site-to-Site como failover automático via BGP

PrivateLink: publicando seu próprio serviço

Interface Endpoints (cap. 4) permitem consumir serviços AWS de forma privada. PrivateLink também permite que você exponha um serviço seu para outras VPCs/contas sem peering:

// Sua VPC (provedor do serviço)
NLB (na frente da sua aplicação)
   ↓
VPC Endpoint Service (associado ao NLB)
   ↓ aprovação manual ou automática de cada subscrição
// VPC do cliente (consumidor)
Interface Endpoint → ENI privada na subnet do cliente → acessa seu serviço por IP privado
💡 Quando usar PrivateLink em vez de Peering Peering expõe toda a CIDR da VPC ao par. PrivateLink expõe apenas o serviço específico atrás do NLB — sem rotas de VPC, sem risco de sobreposição de CIDR entre as VPCs, e com controle granular de quem pode se conectar. Para "SaaS multi-tenant" ou "compartilhar um serviço com parceiros sem expor a rede inteira", a resposta é PrivateLink.

VPC Sharing (AWS RAM)

O AWS Resource Access Manager permite que uma conta "proprietária" compartilhe subnets de sua VPC com outras contas da Organization. As contas participantes podem lançar recursos (EC2, RDS, Lambda em VPC) nessas subnets compartilhadas, mas não podem gerenciar a VPC em si — útil para times de aplicação que não devem administrar a rede, mantendo um único bloco CIDR/Transit Gateway centralizado.

  • Direct Connect Gateway associado a Transit VIFs (conectados a Transit Gateways em cada região) — o DX Gateway estende o alcance da conexão física para múltiplas regiões.
  • Site-to-Site VPN como backup, com BGP configurado para assumir automaticamente as rotas quando o DX cair — ambos anunciam as mesmas rotas, e o roteador escolhe o caminho com menor custo BGP (DX) enquanto disponível.
  • AWS PrivateLink (VPC Endpoint Service na frente de um NLB) — expõe apenas o serviço específico via ENI privada na VPC do cliente, sem peering e sem expor a rede inteira.
  • AWS Network Firewall com regras de filtragem por domínio (stateful), posicionado na rota de saída da VPC — SG/NACL não filtram por FQDN, e WAF só atua em tráfego HTTP de entrada.

💡 SageMaker em uma frase por etapa SageMaker Studio/Notebooks = ambiente de desenvolvimento. SageMaker Training Jobs = treina o modelo em instâncias efêmeras (não paga por instâncias ociosas). SageMaker Endpoints = hospeda o modelo treinado para inferência. SageMaker Pipelines = orquestra o fluxo completo (MLOps) — análogo ao Step Functions, mas para ML.

Amazon Bedrock e o padrão RAG

O Bedrock dá acesso via API a modelos de fundação (foundation models) de múltiplos provedores (Anthropic, Meta, Amazon Titan, etc.) sem provisionar GPU. O padrão mais cobrado é RAG (Retrieval-Augmented Generation):

// Pergunta do usuário não vai direto pro modelo — primeiro busca contexto relevante
Usuário → pergunta
   ↓
Knowledge Base (Bedrock) → busca vetorial nos seus documentos (S3 + OpenSearch/Aurora pgvector)
   ↓
Foundation Model ← pergunta + trechos relevantes recuperados
   ↓
resposta fundamentada nos SEUS dados, sem precisar re-treinar o modelo
★ RAG vs Fine-tuning Se o enunciado diz "as respostas devem refletir documentos internos atualizados frequentemente" — RAG (Knowledge Bases for Bedrock), porque os dados ficam fora do modelo e são buscados em tempo real. Fine-tuning (treinar o modelo com dados próprios) é mais caro, mais lento de atualizar, e raramente é "a resposta certa" quando a palavra-chave é "atualizado"/"dinâmico".

  • Amazon Textract — OCR estruturado que extrai pares chave-valor e tabelas de formulários/documentos, não apenas texto corrido.

CAP. 19

AWS Organizations, Control Tower e governança multi-conta

Consolidar

Empresas reais raramente vivem em uma única conta AWS — usam múltiplas contas para isolar produção de desenvolvimento, separar times e limitar o "blast radius" de erros e incidentes de segurança. Este capítulo cobre como a AWS organiza, governa e fatura essas contas como um todo.

Estrutura do AWS Organizations

ConceitoO que é
Management accountA conta "raiz" da Organization — paga a fatura consolidada e cria/gerencia as demais contas. Não deve hospedar workloads (best practice)
Member accountsContas individuais (produção, dev, segurança, logging...) — isoladas entre si por padrão, cada uma com seu próprio espaço de recursos e IAM
Organizational Units (OUs)"Pastas" que agrupam contas com necessidades de política semelhantes (ex.: OU "Produção", OU "Sandbox") — SCPs aplicadas em uma OU são herdadas por todas as contas dentro dela
RootO nível mais alto da árvore, contém todas as OUs/contas — SCPs aplicadas aqui afetam toda a Organization

Service Control Policies (SCPs)

SCPs definem o teto máximo de permissões para todas as identidades (usuários e roles, incluindo administradores) dentro das contas afetadas — mas, diferente de uma política IAM, uma SCP nunca concede nada por si só.

// SCP "deny-list": bloqueia uma ação específica e herda o resto normalmente
{
  "Effect": "Deny",
  "Action": "ec2:*",
  "Resource": "*",
  "Condition": { "StringNotEquals": { "aws:RequestedRegion": ["sa-east-1"] } }
}
// resultado: ninguém na conta/OU usa EC2 fora de São Paulo — nem mesmo um admin com AdministratorAccess
⚠️ Armadilha: SCP define o teto, não concede acesso Uma SCP do tipo allow-list aplicada no Root (ex.: "Effect": "Allow", "Action": ["s3:*", "ec2:*"]) substitui a política padrão "FullAWSAccess" — a partir daí, nenhuma ação fora dessa lista funciona em conta alguma, mesmo que a política IAM do usuário permita. SCP e política IAM/permission boundary (cap. 1) precisam ambas permitir para a ação ser executada: é uma intersecção de permissões, não uma soma.

AWS RAM e estratégia de tags

AWS Resource Access Manager (RAM) compartilha recursos específicos — subnets de VPC, Transit Gateways, licenças, regras do Route 53 Resolver — entre contas da Organization, sem duplicar o recurso nem montar peering/cross-account IAM para cada caso. Combinado com uma estratégia de tags consistente (ex.: cost-center, environment, owner aplicadas via Tag Policies do Organizations) e Cost Allocation Tags ativadas no Billing, a empresa consegue fatiar o Cost & Usage Report (cap. 13) por equipe ou projeto mesmo dentro de uma fatura consolidada única.

Checkpoint de revisão

  • Uma SCP aplicada à conta ou OU está negando (ou, no caso de allow-list, não incluindo) a ação de S3. SCPs são avaliadas em conjunto com as políticas IAM — se a SCP não permite, nenhuma política IAM consegue sobrepor isso.
  • Um guardrail preventivo do Control Tower (implementado via SCP) negando ações como cloudtrail:StopLogging/cloudtrail:DeleteTrail em todas as contas-membro.
  • Um guardrail detetivo do Control Tower (AWS Config rule) que verifica a configuração de bloqueio de acesso público dos buckets em todas as contas e reporta as não conformidades.
  • Sim, por padrão. Savings Plans (assim como RIs) compartilham o desconto com todas as contas da Organization que tiverem consolidated billing habilitado e uso elegível, a menos que esse compartilhamento seja explicitamente desligado.
  • AWS Resource Access Manager (RAM) — compartilha subnets específicas da VPC com as contas da Organization; as contas participantes lançam recursos nessas subnets, mas a VPC continua gerenciada centralmente.
CAP. 20

AWS Well-Architected Framework

Base

Este é o capítulo "guarda-chuva" da prova: o Well-Architected Framework não introduz serviços novos, mas dá uma lente de leitura para qualquer questão de cenário. Quando o enunciado descreve uma arquitetura e pergunta "qual a MELHOR solução", ele está, na prática, pedindo para identificar qual pilar está sendo testado — e a resposta certa é quase sempre a que resolve esse pilar sem sacrificar demais os outros.

Os seis pilares

PilarFocoPergunta-chave
Operational ExcellenceOperar e monitorar sistemas, melhorando processos continuamente"Como saberemos que algo quebrou, e como automatizamos a resposta?"
SecurityProteger dados, sistemas e ativos via gestão de risco"Quem pode acessar o quê, e os dados estão protegidos em repouso/trânsito?"
ReliabilityRecuperação de falhas, escalonamento dinâmico de demanda"O sistema se recupera sozinho de uma falha de AZ ou instância?"
Performance EfficiencyUso eficiente de recursos computacionais conforme a demanda evolui"Estamos usando o tipo certo de recurso para essa carga de trabalho?"
Cost OptimizationEvitar gastos desnecessários e entender para onde o dinheiro vai"Estamos pagando por capacidade ociosa quando algo serverless/Spot resolveria?"
SustainabilityMinimizar os impactos ambientais das workloads em nuvem"Podemos reduzir consumo de energia escolhendo regiões/instâncias mais eficientes?"
🧠 Mantra: a palavra-chave do enunciado aponta o pilar "Mínimo esforço operacional" → Operational Excellence (pense gerenciado/serverless). "Criptografado", "least privilege", "auditoria" → Security. "Sobrevive à perda de uma AZ", "RPO/RTO" → Reliability. "Mais rápido", "menor latência" → Performance Efficiency. "Mais barato", "reduzir custo" → Cost Optimization. Quando duas opções parecem tecnicamente corretas, o pilar destacado no enunciado decide o desempate.

Checkpoint de revisão

  • Operational Excellence. "Mínimo esforço operacional" é a palavra-chave clássica para soluções serverless/gerenciadas — Lambda + API Gateway tende a ser a resposta, pois elimina patching, capacity planning e gestão de servidores.
  • Cost Optimization. Para workloads tolerantes a interrupção, Spot Instances vencem por custo significativamente menor — RIs fazem sentido para cargas previsíveis e contínuas, não para batch jobs flexíveis.
  • AWS Well-Architected Tool — questionário guiado por workload, sem custo adicional, disponível em qualquer plano de suporte.
  • Não. No plano Basic/Developer, o Trusted Advisor expõe apenas um subconjunto de checks de Security e Service Limits. Cobertura completa das 5 categorias exige plano Business ou Enterprise.
  • Reliability. RPO/RTO numéricos sinalizam que a resposta precisa de uma estratégia de DR concreta (cap. 7) — Multi-AZ ou Multi-Region com backups/replicação que atendam a essas metas, não apenas "fazer backup".
  • Sustainability. Ações típicas: escolher regiões com matriz energética mais limpa, usar instâncias mais eficientes (ex.: Graviton), desligar recursos ociosos e adotar arquiteturas serverless que escalam a zero.
