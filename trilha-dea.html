CAP. 01

Fundamentos de engenharia de dados

Alta prioridade

O vocabulário que o exame pressupõe

ConceitoDefinição em uma linha
ETL vs ELTETL transforma ANTES de carregar (Glue → Redshift). ELT carrega cru e transforma DENTRO do destino (S3 → SQL no Redshift/Athena)
Batch vs StreamingBatch processa lotes em intervalos (Glue jobs). Streaming processa eventos conforme chegam (Kinesis, Flink)
Data Lake vs Data WarehouseLake (S3): dado cru, qualquer formato, schema-on-read. Warehouse (Redshift): estruturado, otimizado para análise, schema-on-write
CDCChange Data Capture: capturar apenas as mudanças de um banco (inserts/updates/deletes) — base do DMS e da replicação contínua
IdempotênciaReprocessar o mesmo dado produz o mesmo resultado — essencial em pipelines com retry
Schema evolutionO schema do dado muda com o tempo; formatos como Avro/Parquet + Glue Schema Registry lidam com isso
Data skewDistribuição desigual de dados entre partições/workers — causa nº 1 de jobs Spark lentos

Formatos de arquivo — a decisão mais repetida do exame

FormatoTipoQuando é a resposta
ParquetColunar, binário, comprimidoQuase sempre. Analytics, Athena, Redshift Spectrum, Glue — lê só as colunas necessárias
ORCColunarEquivalente ao Parquet, mais comum no ecossistema Hive
AvroLinha a linha, binário, com schema embutidoStreaming + schema evolution (Kafka/Kinesis), serialização entre sistemas
JSONLinha a linha, textoIngestão de APIs/eventos — converta para Parquet o quanto antes
CSVLinha a linha, textoIntercâmbio simples — caro e lento para analytics; aparece como "o que substituir"
★
O mantra do DEAParquet + particionamento + compressão = menos dado escaneado = menos custo + mais performance. Esta é a resposta de otimização padrão em Athena, Redshift Spectrum, Glue e EMR. Quando duas alternativas parecem certas, a que envolve formato colunar e partição costuma vencer.

Compressão — o detalhe que derruba candidatos

CodecSplittable?Característica
SnappySim (dentro de Parquet/ORC)Rápido, taxa moderada — o padrão para analytics
gzipNãoBoa taxa, mas um arquivo .gz gigante não pode ser dividido entre workers → gargalo de paralelismo
bzip2SimAlta taxa, lento — raro
LZOSim (indexado)Ecossistema Hadoop
⚠
Armadilha de splittability"Um único arquivo CSV de 50 GB comprimido em gzip está lento no processamento" → o problema é que gzip não é divisível: um único worker processa tudo. Solução: dividir em vários arquivos e/ou usar formato colunar com Snappy.

Particionamento estilo Hive

s3://datalake/vendas/ano=2026/mes=06/dia=12/arquivo.parquet
                      └── filtros WHERE ano/mes/dia só leem a pasta certa (partition pruning)

Particione pelas colunas mais usadas em filtros (geralmente data). Cuidado com cardinalidade alta demais: particionar por cliente_id com milhões de clientes cria milhões de pastas minúsculas — o problema dos small files.

Checkpoint de revisão

  • Converter para Parquet (colunar — lê só as colunas necessárias) e particionar por data (partition pruning — lê só as pastas filtradas). Compressão Snappy complementa.
  • Avro — schema embutido e evolução de schema nativa, idealmente com Glue Schema Registry para validar compatibilidade.
  • ETL: Glue/EMR transforma antes de gravar no destino. ELT: carrega cru no S3/Redshift e transforma lá dentro com SQL (CTAS, dbt, stored procedures).
CAP. 02

Ingestão streaming: Kinesis e MSK

Alta prioridade

A decisão central: Data Streams × Firehose × MSK

Kinesis Data StreamsData FirehoseMSK
O que éStream bruto: você produz e consomeServiço de entrega: recebe e deposita no destinoApache Kafka gerenciado
LatênciaTempo real (<1s)Near real-time (buffer mínimo de segundos)Tempo real
ConsumidoresMúltiplos, com replay (retenção 24h–365d)Nenhum — entrega direto no destinoMúltiplos, com replay
GestãoShards (ou on-demand)Zero — totalmente gerenciadoBrokers/tópicos (ou MSK Serverless)
Gatilho na questão"processar em tempo real", "múltiplos consumidores", "reprocessar""só entregar no S3/Redshift sem gerenciar nada""já usa Kafka", "compatibilidade Kafka"
⚠
A pegadinha mais clássica do DEAFirehose NÃO é tempo real — é near real-time (buffer por tamanho ou tempo). Se a questão exige latência de sub-segundo para o consumidor, a resposta é Kinesis Data Streams (ou MSK), nunca Firehose.

Kinesis Data Streams a fundo

ConceitoO que saber para a prova
ShardUnidade de capacidade: 1 MB/s ou 1.000 registros/s de escrita; 2 MB/s de leitura. Escalar = resharding (split/merge)
Partition keyDefine para qual shard o registro vai. Chave mal distribuída → hot shard → ProvisionedThroughputExceeded
Capacity modesOn-demand (escala sozinho, carga imprevisível) vs Provisioned (mais barato para carga estável)
Enhanced fan-out2 MB/s dedicados por consumidor via push — resposta para "vários consumidores com baixa latência"
Retenção24h padrão, extensível até 365 dias — permite replay/reprocessamento
Producers/ConsumersKPL (agregação, mais throughput), SDK, Kinesis Agent | KCL (checkpointing em DynamoDB), Lambda
⚠
Erro ProvisionedThroughputExceededCausas e soluções em ordem: (1) hot partition → melhorar a distribuição da partition key; (2) capacidade insuficiente → aumentar shards ou mudar para on-demand; (3) no producer → retries com backoff exponencial e agregação com KPL.

Data Firehose a fundo

Origem: Direct PUT, Kinesis Data Streams, MSK. Destinos: S3, Redshift, OpenSearch, HTTP endpoints e parceiros. Recursos cobrados em prova: buffer hints (tamanho MB ou intervalo em segundos — o que vier primeiro), transformação via Lambda em trânsito, conversão de formato nativa para Parquet/ORC (exige schema no Glue Catalog), particionamento dinâmico na escrita do S3, e backup de registros com falha em um bucket de erro.

Processamento de stream: Managed Service for Apache Flink

Para analisar/transformar o stream em movimento (janelas de tempo, agregações em tempo real, detecção de anomalias) com SQL ou Java/Python — antigo Kinesis Data Analytics. Gatilho: "agregações por janela de 1 minuto sobre o stream", "detectar padrão em tempo real". Lambda serve para transformações simples por registro; Flink para lógica com estado e janelas.

🧪
Lab — Streaming de ponta a ponta (1h30, ~US$ 0,10)

    Crie um Firehose com destino S3 e buffer de 60 segundos. Envie JSONs fake de "pedidos" com um script Python usando boto3 put_record.

  2. 02

    Ative a conversão para Parquet (crie antes uma tabela simples no Glue Catalog com o schema). Compare o tamanho dos arquivos JSON vs Parquet no S3.

  3. 03

Checkpoint de revisão

  • Data Firehose com conversão de formato para Parquet. Zero gestão, entrega gerenciada, buffer configurável.
  • Kinesis Data Streams com Enhanced Fan-Out — 2 MB/s dedicados por consumidor via push, sem competir pelo throughput compartilhado.
  • Hot shard — a partition key está concentrando tráfego em um shard. Solução: usar uma chave com melhor distribuição (ex.: adicionar sufixo aleatório).
  • Managed Service for Apache Flink — processamento com estado e janelas de tempo sobre o stream.
CAP. 03

Ingestão batch e migração de dados

Alta prioridade

Tabela de decisão — de onde vem o dado?

Origem do dadoServiçoDetalhe de prova
Banco de dados relacional (contínuo ou one-off)DMSCDC para replicação contínua; full load + CDC para migração sem downtime; SCT se mudar de engine
Arquivos NFS/SMB/HDFS on-premises → S3DataSyncAgendável, verificação de integridade, filtros, throttling de banda
Aplicações SaaS (Salesforce, SAP, Google Analytics)AppFlowConectores prontos, transformações simples, agendado ou por evento
Parceiros enviando arquivos via SFTP/FTPSTransfer FamilySFTP gerenciado sobre o S3
Volumes enormes com rede lentaSnow FamilySe pela rede levar mais de ~1 semana → Snowball
APIs e eventos pontuaisLambda / API GatewayIngestão leve orientada a evento
Aurora → Redshift sem pipelineZero-ETL integrationReplicação gerenciada e contínua, sem Glue/DMS — gatilho: "sem construir pipeline"
⚠
DMS vs DataSync vs AppFlowDMS = bancos de dados (linhas/tabelas, CDC). DataSync = arquivos (NFS/SMB/objetos). AppFlow = SaaS. A questão sempre revela a origem — basta classificar.

S3 como porta de entrada — recursos de ingestão

S3 Event Notifications → EventBridge/SQS/Lambda: o padrão "chegou arquivo, dispara pipeline". Multipart Upload para arquivos grandes. S3 Batch Operations: aplicar uma operação (copiar, etiquetar, invocar Lambda) sobre bilhões de objetos existentes — gatilho: "processar todos os objetos já existentes no bucket".

Checkpoint de revisão

  • DMS com CDC (Change Data Capture), destino S3. Full load inicial + CDC contínuo.
  • Amazon AppFlow — conector nativo do Salesforce, agendamento e mapeamento de campos sem código.
  • Integração zero-ETL Aurora → Redshift — replicação contínua gerenciada pela AWS.
CAP. 04

AWS Glue — ETL e Data Catalog

Alta prioridade

As peças do Glue — não confunda os papéis

ComponentePapelDetalhe de prova
Data CatalogCatálogo central de metadados (databases, tabelas, schemas, partições)Usado por Athena, Redshift Spectrum, EMR e Lake Formation — é o "índice" do data lake
CrawlerVarre a fonte (S3, JDBC), infere o schema e cria/atualiza tabelas no CatalogAgendável; detecta novas partições; classifiers customizados para formatos exóticos
JobO ETL em si — Spark serverless (ou Python Shell para tarefas leves)Pago por DPU-hora; Glue 4.0+; auto scaling de workers
Job BookmarkLembra o que já foi processado → execuções incrementaisGatilho: "processar apenas dados novos a cada execução"
Trigger / WorkflowOrquestração nativa de crawlers e jobs em DAGPara pipelines 100% Glue; além disso → Step Functions/MWAA
Glue Data QualityRegras de qualidade (DQDL) sobre tabelas e dentro de jobsGatilho: "validar completude/unicidade antes de carregar"
Schema RegistryVersiona e valida schemas de streaming (Avro/JSON/Protobuf)Gatilho: "evitar que producers quebrem consumers com schema incompatível"
Glue DataBrewPreparação visual de dados, sem código (250+ transformações)Gatilho: "analistas sem conhecimento de programação precisam limpar dados"

Tuning e troubleshooting de jobs (Domínio 3 adora)

SintomaCausa provável → solução
Job lento ou com OOM (out of memory)Poucos workers ou worker pequeno → aumentar número/tipo de workers (G.1X → G.2X); verificar data skew
Job reprocessa tudo a cada execuçãoBookmark desativado → ativar Job Bookmark
Milhares de arquivos pequenos na saídaExcesso de partições Spark → coalesce/repartition antes de gravar; ou compaction posterior
Job lê a tabela inteira mas só precisa de parteUsar pushdown predicates / partition pruning ao ler do Catalog
Diagnóstico profundo de performanceHabilitar Spark UI e métricas de job no CloudWatch
💡
DynamicFrame vs DataFrameDynamicFrame é a abstração do Glue: tolera schema inconsistente (campos com tipos mistos viram "choice types") e tem transformações próprias (ResolveChoice, Relationalize). Converta para Spark DataFrame quando precisar da API completa do Spark.
🧪

  2. 02

Checkpoint de revisão

  • Job Bookmarks — o Glue rastreia o que já foi processado e faz execuções incrementais automaticamente.
  • Data skew — algumas partições concentram dados demais. Mitigar redistribuindo (salting/repartition) e/ou usando workers maiores (G.2X) ou mais numerosos.
  • Glue DataBrew — interface visual com mais de 250 transformações prontas, sem programação.
CAP. 05

Data Lake no S3

Alta prioridade

Arquitetura de zonas

s3://datalake/
  ├── raw/        ← dado cru, como chegou (imutável)
  ├── processed/  ← limpo, Parquet, particionado
  └── curated/    ← agregado, pronto para consumo/BI

Otimização de arquivos — o problema dos small files

⚠
Small files problemMilhões de arquivos de poucos KB matam a performance de Athena/Spark/EMR — o overhead de abrir cada arquivo supera o tempo de leitura. Tamanho ideal: ~128 MB a 1 GB por arquivo. Soluções: aumentar buffer do Firehose, compaction periódica (job Glue/Spark com coalesce, ou compaction automática de tabelas Iceberg).

Formatos de tabela transacionais (lakehouse)

TecnologiaO que adiciona ao data lake
Apache IcebergTransações ACID, time travel, schema evolution, upserts/deletes no S3 — suportado por Athena, Glue, EMR e Redshift. O preferido da AWS e da prova
Apache Hudi / Delta LakeMesma categoria — aparecem como alternativas, principalmente no EMR
💡
Gatilho de prova"Precisamos fazer UPDATE/DELETE em registros no data lake S3" ou "atender requisições de exclusão de dados pessoais (LGPD/GDPR) no lake" → formato de tabela transacional (Iceberg) — arquivos Parquet puros são imutáveis.

Gestão de armazenamento e custo

Tudo do S3 que vale para o SAA vale aqui: classes de armazenamento, lifecycle rules (raw → IA/Glacier após X dias), Intelligent-Tiering para padrão desconhecido, versionamento, S3 Inventory (relatório de objetos para auditar o lake) e Storage Lens (visibilidade de uso). Para dados frequentemente consultados, mantenha em Standard; arquive zonas raw antigas.

Checkpoint de revisão

  • Compaction: job periódico que reescreve os arquivos pequenos em arquivos de ~128 MB–1 GB; e aumentar o buffer do Firehose para gerar arquivos maiores daqui em diante.
  • Migrar/gerenciar a tabela com Apache Iceberg (ou Hudi/Delta) — formatos transacionais permitem DELETE em nível de linha no data lake.
CAP. 06

Amazon Redshift

Alta prioridade

Carga e descarga de dados

★
COPY é a respostaPara carregar dados no Redshift, o jeito certo é o comando COPY do S3 — paralelo, em massa. INSERTs linha a linha são sempre a alternativa errada. Na direção contrária, UNLOAD exporta resultados de query para o S3 (em Parquet!).

Distribution styles e sort keys — performance de tabela

EstiloComo distribuiQuando usar
KEYLinhas com o mesmo valor da coluna ficam no mesmo nóColuna usada em JOINs frequentes — evita movimentação de dados entre nós
ALLCópia completa da tabela em todos os nósTabelas pequenas de dimensão que entram em muitos JOINs
EVENRound-robin uniformeSem coluna de JOIN clara; tabelas grandes sem padrão
AUTORedshift decide e ajustaDefault moderno — boa resposta quando "menor esforço"

Sort keys: ordenam fisicamente os dados — filtros por data (a sort key mais comum) leem só os blocos relevantes. VACUUM reorganiza/recupera espaço após deletes; ANALYZE atualiza estatísticas do otimizador.

Recursos que caem em prova

RecursoGatilho na questão
Redshift Spectrum"Consultar dados no S3 a partir do Redshift sem carregar" — usa o Glue Catalog
Redshift Serverless"Carga intermitente/imprevisível, pagar pelo uso (RPUs), sem gerenciar cluster"
Concurrency Scaling"Picos de usuários simultâneos em dashboards" — clusters extras transparentes e temporários
WLM (Workload Management)"Separar/priorizar filas: ETL pesado vs queries de BI" — automatic WLM é o default recomendado
Materialized views"Dashboard repete a mesma agregação pesada" — pré-computa e atualiza incrementalmente
Federated query"Consultar RDS/Aurora direto do Redshift, sem mover dado"
Data sharing"Compartilhar dados ao vivo entre clusters/contas sem copiar"
Streaming ingestion"Ingestão direta de Kinesis/MSK no Redshift com latência de segundos" (via materialized view)
Snapshots cross-regionDR do warehouse

CAP. 07

Amazon Athena

Alta prioridade

O modelo: SQL serverless, pago por dado escaneado

Athena consulta o S3 diretamente usando as tabelas do Glue Catalog. A cobrança é por TB escaneado — toda a otimização do Athena gira em torno de escanear menos.

Como escanear menos (e pagar menos)

TécnicaEfeito
Formato colunar (Parquet/ORC)SELECT de 3 colunas lê só essas 3 colunas — redução de 90%+ vs CSV
Particionamento + WHERE na partiçãoPartition pruning: lê só as pastas filtradas
CompressãoMenos bytes lidos = menos cobrado
SELECT colunas específicasNunca SELECT * em produção
Partition projectionCalcula partições por regra (sem crawler) — resolve "tabelas com milhões de partições deixam o planejamento lento"
CTASCREATE TABLE AS SELECT: materializa resultado já em Parquet particionado — também serve para converter formatos via SQL

Recursos de prova

Workgroups: separar times, aplicar limites de bytes escaneados por query (controle de custo) e métricas por grupo. Federated queries: consultar fontes fora do S3 (RDS, DynamoDB, CloudWatch Logs) via conectores Lambda. Suporte nativo a Apache Iceberg (incluindo UPDATE/DELETE e time travel). Resultados sempre gravados em um bucket de resultados.

🧪
Lab — Vendo a economia do Parquet (1h30, ~US$ 0,05) ⭐
  1. 01

    No Athena, rode o mesmo SELECT na tabela CSV (raw) e na Parquet (processed) do lab de Glue. Compare o campo "data scanned" — você verá a economia em números reais.

  2. 02

    Filtre pela partição (WHERE mes = '06') e veja o scan cair ainda mais.

  3. 03

    Use CTAS para criar uma tabela agregada em Parquet particionado.

  4. 04

    Crie um workgroup com limite de 100 MB por query e veja o Athena abortar uma query que estoura o limite.

Checkpoint de revisão

  • Workgroups com data usage controls — limite de bytes por query e por workgroup.
  • Partition projection — o Athena calcula as partições por regra, sem consultar o Catalog nem precisar de crawler para cada nova partição.
  • Athena CTAS — CREATE TABLE AS SELECT com format = PARQUET e particionamento na própria query.
CAP. 08

Amazon EMR

Alta prioridade

Anatomia do cluster

NóPapelPode ser Spot?
Primary (master)Coordena o clusterNão recomendado
CoreProcessa E armazena (HDFS)Arriscado — perder core = perder dado do HDFS
TaskSó processa, sem HDFSSim! A resposta clássica de custo

Use EMRFS (dados no S3) em vez de HDFS para desacoplar storage de compute — clusters transientes que sobem, processam e morrem são o padrão de custo. EMR Serverless: submeter jobs Spark/Hive sem dimensionar cluster nenhum. EMR on EKS: rodar Spark no cluster Kubernetes existente.

Glue vs EMR vs Lambda — a decisão dos 34%

Sinal na questãoResposta
"ETL serverless, integrado ao Catalog, mínimo esforço operacional"Glue
"Controle do cluster, bibliotecas customizadas, frameworks além de Spark (Hive, Presto, HBase, Flink)"EMR
"Jobs Spark esporádicos sem dimensionar cluster"EMR Serverless
"Transformação leve, por evento, menos de 15 minutos"Lambda
"Migrar cluster Hadoop on-premises mantendo o ecossistema"EMR

Checkpoint de revisão

  • Spot Instances nos task nodes (e cluster transiente com dados no S3 via EMRFS, terminando após o job). Primary e core em On-Demand.
  • EMR — controle total do ambiente, bootstrap actions para instalar dependências.

Orquestração de pipelines

Alta prioridade

Quem coordena o quê

ServiçoModeloGatilho na questão

💡
Step Functions: Standard vs ExpressStandard: até 1 ano de duração, exactly-once, auditável passo a passo — pipelines de dados. Express: alto volume, curto (até 5 min), at-least-once, mais barato — ingestão de eventos em massa.

O padrão completo que a prova adoraS3 Event → EventBridge → Step Functions (inicia Glue Job → valida qualidade → Choice: sucesso publica no SNS / falha vai para o fluxo de erro com retry e alerta). Saber montar mentalmente essa cadeia resolve várias questões do Domínio 3.

⚠
Métrica de prova: IteratorAgeNo Kinesis/Lambda, IteratorAge alta = consumidor não está dando conta — os registros estão envelhecendo no stream. Soluções: aumentar paralelismo (mais shards, parallelization factor da Lambda), aumentar batch size, otimizar o processamento.

Qualidade de dados

Glue Data Quality (regras DQDL: completude, unicidade, faixas de valores) é a resposta gerenciada — pode rodar sobre tabelas do Catalog ou dentro de jobs, e travar o pipeline se a qualidade reprovar. A prova também espera o conceito de quarentena: registros reprovados vão para um caminho separado no S3 em vez de contaminar a zona processed.

Idempotência e reprocessamento

Pipelines reais falham e re-executam — o exame cobra os mecanismos: Job Bookmarks (Glue), checkpoints (Flink/KCL), DLQs (SQS/Lambda), escrita idempotente (sobrescrever partição inteira em vez de append duplicado) e exactly-once vs at-least-once (consumidor deve tolerar duplicatas em sistemas at-least-once).

Checkpoint de revisão

  • Glue Data Quality com regra de completude/não-nulo no pipeline; resultado FAILED roteia os registros para quarentena e dispara SNS via EventBridge.
  • O consumidor está atrasado em relação ao stream. Escalar o consumo: mais shards, aumentar o parallelization factor da Lambda, ou otimizar o tempo de processamento por batch.
CAP. 11

Segurança e governança de dados

Alta prioridade

Lake Formation — o centro da governança

O IAM controla acesso a serviços e recursos; o Lake Formation controla acesso a dados do lake com granularidade que o IAM sozinho não alcança: database, tabela, coluna, linha e célula. Registre as localizações do S3, conceda permissões aos principals (estilo GRANT/REVOKE) e todos os serviços integrados (Athena, Redshift Spectrum, Glue, EMR) respeitam as regras.

Column-level security"Analistas podem ver a tabela, mas não a coluna salario"

LF-Tags (TBAC)"Gerenciar permissões em escala etiquetando dados (ex.: tag confidencial) em vez de conceder tabela por tabela"
Cross-account sharing"Compartilhar tabelas do Catalog com outra conta de forma governada"
